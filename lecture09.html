Introduction to Cypher
Graph Query Language

Cypher is Neo4j's query language for graph databases. Learn how to Create, Read, Update, and Delete nodes and relationships — plus filtering, pipelines, and loading CSV data.

🏠 Home ← Lecture 07
🎯 Exam & Project Importance

01

What is Cypher?

📖 Definition

Cypher is the query language for Neo4j — the most popular graph database. It is designed to be easy to read and write, resembling plain English. Instead of selecting from tables (like SQL), Cypher describes a pattern you want to find in the graph.

Think of it this way: SQL says "select from table." Cypher says "find me things that look like this pattern." You describe the nodes and relationships you want, and Neo4j finds them.

💬 Lecturer's Analogy

"How do you describe a graph? Just use your natural language. How many nodes do we have? How many relationships? Any properties? Once you can describe it in English, you can write it in Cypher."

Cypher vs SQL — The Big Picture

SQL (Relational DB)
  • SELECT — query data
  • INSERT — add data
  • UPDATE — modify data
  • DELETE — remove rows
  • DROP — remove whole table
Cypher (Neo4j)
  • MATCH + RETURN — query data
  • CREATE or MERGE — add data
  • SET — modify properties
  • DELETE / DETACH DELETE — remove nodes/rels
  • REMOVE — remove a property
🎯 Key Difference from SQL

In Cypher there is no GROUP BY, no HAVING, no subqueries, no window functions, and limited aggregation functions (only COUNT, SUM, AVG, MIN, MAX). Cypher is for traversing relationships — use SQL for complex aggregations.

02

Cypher Syntax Basics

Before writing queries, you need to understand how nodes and relationships are written in Cypher. It uses ASCII art — symbols that visually look like a graph.

Representing Nodes — Round Brackets ( )

Nodes are always written inside round brackets. The general format is:

Node syntax
(variable:Label {property: "value"})
( )= round brackets → this is a NODE p1= variable (optional name you can use later in the query) :Person= label (the type of node, like a table name) {name:"Siri"}= properties (key-value pairs, like attributes)

Real Examples from the slides:

Node examples
// Bare node — no label, no properties (n) // Node with a label only (:Person) // Node with label and one property (:Person {name: "Siri"}) // Node with label and multiple properties (:Person {name: "Siri", gender: "F"}) // Node with variable + label + property (most common in queries) (p1:Person {name: "Siri"})

Representing Relationships — Square Brackets [ ]

Relationships are always written inside square brackets. Direction is shown using hyphens and arrows.

Relationship syntax
[variable:LABEL {property: "value"}]
[ ]= square brackets → this is a RELATIONSHIP :WORKS_FOR= relationship type (name of the relationship) {since: 2018}= relationship properties (optional)

Real Examples:

Relationship examples
// Relationship with no properties [:WORKS_FOR] // Relationship with a property [:IS_FRIEND_WITH {since: 2018}] // Relationship with a variable (so you can reference it later) [r:READ]

Connecting Nodes and Relationships — The Arrow

You connect nodes and relationships using hyphens - and a greater-than sign > to show direction. Direction matters — it changes the meaning entirely.

Full pattern syntax
// Jennifer works FOR Neo4j (direction: Person → Company) (p:Person {name: "Jennifer"})-[:WORKS_FOR]->(c:Company) // The movie was DIRECTED BY a person (arrow points towards movie) (:Movie)<-[:DIRECTED]-(p:Person)
🎯 Direction Matters — Exam Trap

The lecturer emphasised: "The direction is very important in Neo4j because it tells the database which way the relationship follows. If you change the direction, the meaning is totally different."
(:Person)-[:WORKS_FOR]→(:Company) = Person works for Company
(:Company)-[:WORKS_FOR]→(:Person) = completely wrong meaning

Three Things Inside a Node or Relationship

Thing What it is Optional? Example
Variable A name you give to reference the node/relationship later in the same query Yes p in (p:Person)
Label The type of the node (like a table name). Must have a colon before it. Yes :Person, :Movie, :Company
Property Key-value attribute inside curly braces. Strings need quotes; numbers do not. Yes {name: "Siri"}, {age: 28}
03

CREATE — Adding Data

📖 What CREATE does

CREATE adds new nodes or relationships to the database. It always creates something new — even if that node or relationship already exists. (Use MERGE if you want to avoid duplicates.)

Creating a bare node (no label, no properties)

CREATE bare node
CREATE (n)

This creates one empty node. Neo4j automatically assigns it an internal <id>. If you delete and recreate it, the <id> will change — so don't rely on it. Use your own surrogate key.

Creating a node with a label and property

CREATE node — with label and property
CREATE (p:Person {name: "Siri"})
✅ In plain English: Create a new Person node with the name "Siri".
Result: "Created 1 node, set 1 property, added 1 label."

Creating multiple nodes at once

CREATE multiple nodes
CREATE (p1:Person {name: "Siri"}), (p2:Person {name: "Jennifer"}), (c:Company {name: "Neo4j"}), (t:Technology {type: "Graphs"})
✅ Creates 4 nodes in one command. Result: "Created 4 nodes, set 4 properties, added 4 labels."

Creating a relationship between EXISTING nodes

⚠️ Important Rule

You cannot create a relationship without nodes. To create a relationship between existing nodes, you MUST first use MATCH to find them, then CREATE the relationship.

CREATE relationship between existing nodes
MATCH (p:Person {name: "Siri"}), (b:Book {name: "Harry Potter"}) CREATE (p)-[r:READ]->(b) RETURN p, r, b
✅ In plain English: Find the Person "Siri" and the Book "Harry Potter", then create a READ relationship from Siri to the book. Return all three.
Note: RETURN is optional here — the CREATE will work even without it.
04

MERGE — The Safe Way to Create

📖 MERGE = "Create if it doesn't exist, use it if it does"

MERGE checks the database first. If the node or relationship already exists → it does nothing (no duplicate). If it doesn't exist → it creates it. This is safer than CREATE for avoiding duplicates.

💡 Easy Way to Remember

CREATE = always creates (risk of duplicates)
MERGE = creates only if it doesn't already exist (no duplicates)

MERGE matching ALL properties

MERGE — won't duplicate if node exists
// Siri already exists with just name. MERGE won't create again. MERGE (:Person {name: "Siri"}) // But Siri with gender:"F" doesn't exist yet → MERGE creates a NEW node MERGE (:Person {name: "Siri", gender: "F"})
⚠️ MERGE Matches ALL Properties

MERGE checks ALL properties listed. If you add even one extra property (like gender), it treats it as a different node and creates a new one. Be careful — if you want to match only on ID, only include the ID in the MERGE clause.

MERGE for a relationship between existing nodes

MERGE — relationship (use MATCH first)
MATCH (p:Person {name: "Siri"}), (b:Book {name: "Harry Potter"}) MERGE (p)-[r:READ]->(b) RETURN p, r, b
✅ If Siri already has a READ relationship to Harry Potter → does nothing. If not → creates the relationship.
05

MATCH & RETURN — Reading Data

📖 How Reading Works in Cypher

MATCH finds a pattern in the graph (like SQL's FROM/WHERE combined). RETURN specifies what to output (like SQL's SELECT). The simplest query is MATCH + RETURN.

Basic MATCH + RETURN

Example 1 — Find companies Jennifer works for
MATCH (p:Person {name: "Jennifer"})-[:WORKS_FOR]->(c:Company) RETURN p, c
✅ In plain English: "Find all companies that Jennifer works for."
MATCH = find the pattern (Person named Jennifer who works for a Company)
RETURN = show me both: the person (p) and the company (c)

Example 2 — Who directed "Wall Street"?

MATCH with reversed direction (arrow pointing left)
MATCH (:Movie{title:'Wall Street'})<-[:DIRECTED]-(p:Person) RETURN p
✅ In plain English: "Find all people who directed Wall Street."
Arrow points ← because the Person DIRECTED the Movie (Person → Movie). The movie is the "anchor point" — Neo4j starts from Wall Street and searches outward for who directed it.
Result: Returns the Person node "Oliver Stone".

The Anchor Point concept

Cypher starts the search from a specific known node (the anchor point) and then traverses outward. In the query above, the Movie "Wall Street" is the anchor — Neo4j finds it first, then finds who directed it. This is why the query is fast — it only searches the subgraph around Wall Street, not the whole database.

Example 3 — Find actors who acted in the same movie

MATCH — two nodes connected through the same middle node
MATCH (p1:Person)-[r1:ACTED_IN]->(m:Movie)<-[r2:ACTED_IN]-(p2:Person) RETURN p1, p2
✅ In plain English: "Find all pairs of actors who acted in the same movie together."
p1 acted in movie m, AND p2 also acted in movie m. Return both actors as a pair.
🎯 Why does this return more rows than nodes?

If Wall Street has 3 actors (A, B, C), RETURN p1, p2 gives: AB, AC, BA, BC, CA, CB = 6 pairs. But there are only 3 nodes. The query returns every ordered combination. To avoid duplicates, use RETURN DISTINCT p1, p2. The graph view removes duplicates; the table view keeps all of them.

Returning specific properties (not the whole node)

Return a property value, not the whole node
MATCH (m:Movie) RETURN m.title // returns just the title, not the whole node

Using the pipe symbol | (means OR for labels)

Pipe = OR — match multiple relationship types
MATCH (:Movie{title:'Wall Street'})<-[:DIRECTED|ACTED_IN]-(p:Person) RETURN p
✅ In plain English: "Find all people who either DIRECTED or ACTED_IN Wall Street."
The | symbol means OR — it matches either relationship type. Returns both directors and actors.
06

WHERE — Filtering Results

📖 What WHERE does

WHERE filters the results of a MATCH — like SQL's WHERE clause. It uses the same boolean operators: AND, OR, XOR, NOT.

WHERE with AND

WHERE with AND — multiple conditions
MATCH (p1:Person)-[r1:ACTED_IN]->(m:Movie)<-[r2:ACTED_IN]-(p2:Person) WHERE p1.name = "Martin Sheen" AND p2.name = "Rob Reiner" RETURN m.title
✅ In plain English: "Find movies where Martin Sheen acted AND Rob Reiner also acted. Return the movie title."

Two ways to filter (both are correct)

Style 1 — filter inside MATCH
MATCH (p1:Person {name:"Martin"})-[:ACTED_IN]->(m)<-[:ACTED_IN]-(p2) RETURN p1, p2

Filters FIRST, then finds pairs. Faster — the database starts with Martin immediately.

Style 2 — separate WHERE clause
MATCH (p1:Person)-[:ACTED_IN]->(m)<-[:ACTED_IN]-(p2) WHERE p1.name = "Martin" RETURN p1, p2

Finds ALL pairs first, then filters. More readable for complex conditions.

Both produce the same result. Lecturer said: "The logic is different, but both are correct."

String matching — STARTS WITH, ENDS WITH, CONTAINS

Cypher has built-in string functions (like SQL's LIKE operator):

String filtering
// Name starts with "Martin" AND ends with "Sheen" WHERE p1.name STARTS WITH "Martin" AND p2.name ENDS WITH "Sheen" // Name contains "artin" anywhere WHERE p.name CONTAINS "artin"
💡 These are equivalent to SQL's LIKE '%Martin%' etc. Use them instead of LIKE in Cypher.

RETURN DISTINCT — removing duplicates

Remove duplicate results
MATCH (p1)-[:ACTED_IN]->(m) RETURN DISTINCT p1.name
💡 DISTINCT removes duplicate rows — same as SQL's SELECT DISTINCT.
07

WITH — Pipelines and Aggregation

📖 What WITH does

WITH passes results from one part of the query to the next part — like a pipeline. It is also used for aggregation (since Cypher has no GROUP BY).

WITH as a pipeline

WITH — pass results between query parts
MATCH (michael:Person {name: "Michael Douglas"})-[:ACTED_IN]->(m:Movie)<-[:ACTED_IN]-(charlie:Person {name: "Charlie Sheen"}) WITH m, michael, charlie, m.title AS movieTitle MATCH (director:Person:Director)-[:DIRECTED]->(m) RETURN movieTitle AS Movie, michael.name AS Actor1, charlie.name AS Actor2, director.name AS Director
✅ In plain English: Part 1: Find movies where both Michael Douglas and Charlie Sheen acted. Part 2: For those same movies, find the director. Return the movie title, both actors, and the director.
WITH connects the two parts. The movie variable m flows from Part 1 into Part 2.

WITH for aggregation (replacing GROUP BY)

WITH + aggregation (Cypher's version of GROUP BY)
MATCH (director:Person:Director)-[:DIRECTED]->(m:Movie) WITH m.title AS title, COUNT(director) AS directorCount RETURN title, directorCount
✅ This is how you do "GROUP BY" in Cypher — using WITH + COUNT. The movie title is the grouping key; directorCount is the aggregation. Cypher does not have GROUP BY as a keyword.
08

SET & REMOVE — Updating Data

SET — add or update a property

📖 How to Update

To update data: MATCH the node first, then SET the property. You can use SET to both add a new property and update an existing one.

SET — add a new property
MATCH (p:Person{name:"Martin Sheen"}) SET p.birthdate = date("1940-08-03") RETURN p
✅ In plain English: "Find Martin Sheen's node and add a birthdate property with value 1940-08-03."
If birthdate already exists → it gets updated. If not → it gets added.
date() converts the string into a proper date data type.

REMOVE — delete a property

REMOVE — delete a property from a node
MATCH (p:Person {name:"Siri"}) REMOVE p.name RETURN p
✅ In plain English: "Find Siri's node and remove the name property."
After this, the node no longer has a name property. (Not NULL — just absent.)
09

DELETE & DETACH DELETE

⚠️ Golden Rule of Deletion in Neo4j

You cannot delete a node that still has relationships. You must delete the relationships first, then delete the node. If you try to delete a node with relationships, Neo4j will give an error: "G1001: Dependent object error — edges still exist."

DELETE a relationship

DELETE — remove a relationship (not the nodes)
MATCH (p:Person {name: "Siri"})-[r:READ]->(b:Book {name: "Harry Potter"}) DELETE r
✅ In plain English: "Find Siri's READ relationship to Harry Potter, and delete the relationship (not the nodes)."
The Siri and Harry Potter nodes still exist after this — only the relationship is removed.

DELETE a node (must remove relationships first)

DELETE — delete a relationship THEN the node
// Step 1: Delete the relationship first MATCH (p:Person{name:"Siri"})-[r:READ]->(b:Book) DELETE r // Step 2: Now delete the node safely MATCH (p:Person{name:"Siri"}) DELETE p

DETACH DELETE — delete node AND all its relationships at once

💡 DETACH DELETE — the shortcut

Instead of deleting relationships first and the node second, use DETACH DELETE — it deletes the node AND automatically removes all its relationships in one command.

DETACH DELETE — delete node + all relationships
MATCH (b:Book {name: "Harry Potter"}) DETACH DELETE b
✅ In plain English: "Delete the Harry Potter book node AND delete all relationships pointing to/from it."

Delete EVERYTHING in the database

⚠️ DANGER — deletes the entire database
MATCH (n) DETACH DELETE n
⚠️ This deletes everything

This matches ALL nodes (n has no label filter) and detach-deletes all of them — so all nodes, all relationships, and all properties are gone. Use this only when starting fresh. Lecturer said: "Use this to clean the environment when you want to start a new project."

10

LOAD CSV — Importing Data

📖 What LOAD CSV does

LOAD CSV reads a CSV file and creates nodes or relationships from each row. This is how you bulk-import data into Neo4j — instead of creating nodes one by one.

How to set up CSV importing

  1. Place your CSV files in the Neo4j import folder (create it if it doesn't exist).
  2. Reference the file using the file:/// prefix in your Cypher query.
  3. Use WITH HEADERS if your CSV has a header row (recommended).
  4. Use AS row to refer to each line in the CSV as a variable named row.
  5. Use row.ColumnName to access specific column values.

LOAD CSV — creating nodes from a CSV

LOAD CSV — create Order nodes from orders.csv
LOAD CSV WITH HEADERS FROM 'file:///orders.csv' AS row MERGE (order:Order {orderID: row.OrderID}) ON CREATE SET order.shipName = row.ShipName
✅ For each row in orders.csv: create an Order node with the orderID from the CSV. If it already exists (MERGE), set the shipName property. This avoids duplicates.

LOAD CSV — creating relationships from a CSV

LOAD CSV — create relationships (must MATCH nodes first)
LOAD CSV WITH HEADERS FROM 'file:///order_products.csv' AS row MATCH (order:Order {orderID: row.OrderID}) MATCH (product:Product {productID: row.ProductID}) MERGE (order)-[r:CONTAINS]->(product) ON CREATE SET r.quantity = toInteger(row.Quantity), r.unitPrice = toFloat(row.UnitPrice)
✅ For each row in the CSV: find the matching Order and Product nodes, then create a CONTAINS relationship between them with quantity and unitPrice as properties. toInteger() and toFloat() convert string values to numbers.
🎯 Project/Exam — ETL with LOAD CSV

The lecturer said: "For Project 2, you must use Cypher to create nodes and relationships. You are NOT allowed to use the GUI import tool — Cypher contributes to your marks."
Key rule: You need separate CSV files for nodes and for relationships. A single "everything" CSV doesn't work well for Neo4j.

Cypher Limitations vs SQL

Cypher is powerful for graph traversal, but it has significant limitations compared to SQL for analytical work. The lecturer listed these explicitly:

Feature SQL Cypher
GROUP BY ✅ Yes — native keyword ❌ No — use WITH + aggregation instead
HAVING ✅ Yes ❌ No — use WITH + WHERE instead
Subqueries ✅ Yes — nested SELECT ❌ No — not supported
Window functions ✅ Yes — SUM OVER, RANK OVER ❌ No
ROLLUP / CUBE ✅ Yes — multi-dimensional analysis ❌ No
Aggregation functions ✅ Many — MEDIAN, VARIANCE, STDEV, etc. ⚠️ Limited — only COUNT, SUM, AVG, MIN, MAX, STDEV
Graph traversal ❌ Slow — complex recursive SQL ✅ Native, fast, simple
Relationship queries ❌ Complex joins ✅ Simple pattern matching
💬 Lecturer's Rule

"For Cypher: use it for relationships, nodes, and traversal. For complex aggregations (GROUP BY, HAVING, window functions, ROLLUP/CUBE) — use SQL."
Cypher is not a replacement for SQL — it's a complement. Use each for what it's good at.

Summary

Lecture 09 — Key Takeaways

  • Cypher is Neo4j's query language — designed to read like plain English. It describes patterns.
  • Nodes use round brackets ( ) — with label after colon (:Person) and properties in curly braces {name:"Siri"}.
  • Relationships use square brackets [ ] — with relationship name after colon ([:WORKS_FOR]) and direction via arrows → or ←.
  • Variables (p, c, m, r) are optional — they let you reference nodes/relationships later in the same query.
  • Direction matters — (:Person)-[:WORKS_FOR]→(:Company) ≠ (:Company)-[:WORKS_FOR]→(:Person).
  • CRUD in Cypher: CREATE/MERGE → MATCH+RETURN → SET/REMOVE → DELETE/DETACH DELETE.
  • MERGE = "create if not exists." It checks ALL specified properties before deciding.
  • Cannot delete a node with relationships — delete relationships first, or use DETACH DELETE.
  • Cypher has NO GROUP BY, NO HAVING, NO subqueries, NO window functions, NO ROLLUP/CUBE.
  • Use WITH for pipelines and as a workaround for GROUP BY (WITH + aggregation function).
  • LOAD CSV imports data from CSV files. Must use MATCH before creating relationships in CSV import.
  • Use SQL for complex aggregations. Use Cypher for graph traversal and relationship queries.

Exam-Style Practice Questions

Write a Cypher query to find all movies that Tom Hanks acted in. Return the movie titles.

Hint: MATCH (p:Person {name:"Tom Hanks"})-[:ACTED_IN]→(m:Movie) RETURN m.title

Explain the difference between CREATE and MERGE in Cypher. When would you use MERGE over CREATE?

Hint: CREATE always creates (risk of duplicates). MERGE checks first — if node/relationship exists, uses it; if not, creates it. Use MERGE when loading CSV data or when duplicates would be a problem.

You try to delete a Person node "Siri" using DELETE but get an error. Why? How do you fix it?

Hint: Error = "edges still exist." Siri has relationships. Fix: (1) Delete the relationships first, then DELETE Siri. OR (2) Use DETACH DELETE which removes both the node and all its relationships in one command.

What is wrong with this query? Explain and fix it:
MATCH (p:Person {name:"Alice"}) CREATE (p)-[:WORKS_FOR]->(c:Company {name:"UWA"})

Hint: This creates a NEW Company node "UWA" every time it runs — even if UWA already exists. Fix: Use MERGE instead of CREATE for the Company, or MATCH the existing Company node first: MATCH (c:Company {name:"UWA"}) MATCH (p:Person {name:"Alice"}) MERGE (p)-[:WORKS_FOR]->(c)

Write Cypher to find pairs of actors who both acted in the same movie, but only return each unique pair once (no duplicates like AB and BA).

Hint: MATCH (p1:Person)-[:ACTED_IN]→(m:Movie)←[:ACTED_IN]-(p2:Person) WHERE p1.name < p2.name RETURN DISTINCT p1.name, p2.name. Using p1.name < p2.name ensures each pair appears once.

Cypher does not have GROUP BY. How do you count how many movies each director has directed?

Hint: MATCH (d:Person)-[:DIRECTED]→(m:Movie) WITH d.name AS director, COUNT(m) AS movieCount RETURN director, movieCount. The WITH + COUNT replaces GROUP BY. The variable before COUNT (d.name) acts as the grouping key.

You have a CSV file "students.csv" with columns: StudentID, Name, Major. Write Cypher to load it and create Student nodes.

Hint: LOAD CSV WITH HEADERS FROM 'file:///students.csv' AS row MERGE (s:Student {studentID: row.StudentID}) ON CREATE SET s.name = row.Name, s.major = row.Major

What does the pipe symbol | mean in Cypher? Give an example.

Hint: | means OR for relationship types. MATCH (:Movie)←[:DIRECTED|ACTED_IN]-(p:Person) matches people who either DIRECTED or ACTED_IN a movie. Not "and" — either/or.
