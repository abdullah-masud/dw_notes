Closed Patterns and Max-Patterns

Long patterns generate astronomically many sub-patterns (2100 for a 100-item set). Closed and max patterns are compressed representations.

TypeDefinitionProperty
Closed PatternFrequent itemset X where NO super-pattern Y âŠƒ X has the SAME support as XLossless â€” all frequent itemset supports can be derived from closed patterns
Max-PatternFrequent itemset X where there is NO FREQUENT super-pattern Y âŠƒ XLossy â€” we know all subsets are frequent but don't know their exact supports
ðŸŽ¯ Key Relationships

Every max-pattern IS a closed pattern. Not every closed pattern is a max-pattern (a closed pattern may have frequent super-patterns). In the Venn diagram: Frequent âŠƒ Closed âŠƒ Max.

How to Identify Closed Patterns

For each frequent itemset X: check if any of X's supersets have the EXACT SAME support. If yes â†’ X is NOT closed (because adding more items doesn't reduce support). If no â†’ X IS closed.

How to Identify Max-Patterns

For each frequent itemset X: check if any of X's supersets are ALSO frequent. If yes â†’ X is NOT a max-pattern (it can be extended and still be frequent). If no â†’ X IS a max-pattern.

ðŸ’¡ Simple Example

All supports: a=2, b=2, c=1, {a,b}=2, {a,c}=1, {b,c}=1, {a,b,c}=1.
Closed patterns: {a,b} (support=2, superset {a,b,c} has support=1 â€” different, so {a,b} is closed) and {a,b,c} (no supersets â†’ closed).
Max-pattern: Only {a,b,c} (its only supersets don't exist â†’ it's the only max-pattern).
Note: {a,b} is closed but NOT max (because {a,b,c} is a frequent superset).

ðŸŽ¯ Why Prefer Closed Over Max?

Closed patterns are lossless â€” from closed patterns, you can recover the exact support of any frequent itemset. Max-patterns are lossy â€” you only know all subsets are frequent, not their exact supports. In practice, prefer closed patterns for richer information.

The Apriori Algorithm

ðŸ“– The Downward Closure Property (Anti-monotonicity)

If an itemset X is FREQUENT, then ALL subsets of X are also frequent.
Equivalently: if any SUBSET of X is INFREQUENT, then X cannot be frequent.
This is the key insight that makes Apriori efficient â€” prune entire branches early.

Apriori Algorithm Steps

  1. Scan the database to find all frequent 1-itemsets (Fâ‚).
  2. Generate candidate (k+1)-itemsets from frequent k-itemsets by self-joining.
  3. Prune candidates: if ANY subset of a candidate is infrequent, remove it.
  4. Scan database to count support of remaining candidates.
  5. Repeat until no new frequent itemsets are found.
ðŸ’¡ The Pruning Power

If item D is infrequent (say support = 1 when minsup = 2), then ALL itemsets containing D are automatically infrequent. You never need to check AD, BD, CD, ACD, BCD, etc. This dramatically reduces computation.

Self-Joining for Candidate Generation

F3 = {abc, abd, acd, ace, bcd}. To generate C4: join F3 with itself. Only join sets that share the first (k-1) items. Example: {abc} and {abd} â†’ {abcd} (share "ab"). Then prune: {abcd} is pruned if any 3-subset is not in F3. Check: is {acd} in F3? Yes. Is {bcd} in F3? Yes. So {abcd} survives. Result: C4 = {abcd}.

Confidence

Confidence(X â†’ Y) = Support(X âˆª Y) Ã· Support(X) = Support_count(X âˆª Y) Ã· Support_count(X)

Confidence measures: "Of all transactions containing X, what fraction ALSO contain Y?" It is a conditional probability.

Strong Association Rules

A rule is strong if it meets BOTH minimum support AND minimum confidence thresholds. These are the rules we care about.

A high-confidence rule isn't always meaningful. If item B is already very popular (appears in 90% of transactions), then almost any rule "X â†’ B" will have high confidence â€” not because X causes B, but just because B is everywhere. That's why we also need Lift.
