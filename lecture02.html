Fact Tables

ðŸ“– Definition

A fact table is the central table in a data warehouse. It stores measurements (facts/measures) about a business process, and contains foreign keys linking to dimension tables.

Structure of a Fact Table

Every fact table row contains exactly three types of columns:

  • Primary Key (PK): A single surrogate column that uniquely identifies each row.
  • Foreign Keys (FKs): One FK for each dimension table â€” these are numerical references. They CAN be duplicate (same location can have many sales).
  • Measures: Numerical values (e.g., units_sold, dollars_sold, avg_sales). All values in a fact table must be numerical.
ðŸŽ¯ Exam Trap â€” Only Numerical in Fact Table

If you see gender = "M" or "F" in a fact table, that is WRONG. Convert to numbers: 1 = Female, 2 = Male, 3 = Unknown. All values in a fact table must be numerical. Column names are metadata â€” the data must be numbers.

Properties of Fact Tables

  • Very big: Often millions or billions of rows
  • Narrow: Small number of columns (PK + FKs + a few measures)
  • Grows over time: New events â†’ new rows appended

The Grain of a Fact Table

ðŸ“– Definition â€” The Grain

The grain = the meaning of ONE row in the fact table. It defines the most detailed level of data stored. Example: "One row = one line item from a cash register receipt."

Rule of thumb: err in favour of the finest grain possible. Store as much detail as you can. If you store only daily totals, you can never answer "what happened at 9am?" But if you store individual transactions, you can always aggregate up. Cloud storage is cheap â€” detail is valuable.

Factless Fact Tables

ðŸ“– Definition

A factless fact table has no numeric measurements â€” it only records that an event occurred. The measure column always = 1 (event happened). There is NO 0 â€” if the event didn't happen, there is no row.

Example: Recording lecture attendance. Each row = one student at one lecture. The measure = 1 (they attended). If they didn't attend, no row exists.

Dimension Tables

ðŸ“– Definition

A dimension table describes the context of a measurement. It stores descriptive attributes about a real-world object or concept (Customer, Product, Date, Region, Store, etc.).

Properties of Dimension Tables

  • Wide: Many descriptive columns (dozens of attributes)
  • Small: Relatively few rows compared to fact tables (usually <1 million)
  • Static: Contents don't change often â€” almost like a lookup table
Facts
  • Narrow (few columns)
  • Huge (many rows)
  • Numerical values only
  • Grows constantly
Dimensions
  • Wide (many columns)
  • Small (few rows)
  • Descriptive text values
  • Relatively static

Uses of Dimension Tables

  • Filters: WHERE location = 'Perth' â€” uses location dimension
  • Grouping: GROUP BY product_category â€” uses product dimension
  • Reference: Fact table rows point TO dimension rows via FKs

The Three DW Schemas

All data warehouses use one of these three schema designs. You must know all three, their differences, and when each is preferred.

1. Star Schema

ðŸ“– Definition

A star schema has ONE central fact table connected to multiple dimension tables. The shape looks like a star. This is the most common and preferred design in modern DW.

  • Fewer tables â†’ faster queries (fewer joins)
  • Dimension tables may have some redundant data
  • Easier to understand for business users
  • Modern recommendation: use star schema

2. Snowflake Schema

ðŸ“– Definition

A snowflake schema normalises some dimension tables by splitting them into sub-dimension tables. Looks like a snowflake branching from the fact table.

  • Reduces data redundancy in dimension tables
  • More tables â†’ slower queries (more joins required)
  • Harder to understand â€” more complex
  • Example: location dimension splits into location + city sub-dimension

3. Galaxy Schema (Fact Constellation)

ðŸ“– Definition

A galaxy schema has multiple fact tables that share dimension tables. Used when the DW needs to support multiple related business processes (e.g., Sales and Shipping).

  • Most complex schema â€” used in large organisations
  • Different departments may use different fact tables
  • Shared dimensions ensure consistency across processes

ðŸŽ¯ Exam Point â€” Modern Design Preference

Despite the snowflake schema reducing redundancy, star schema is the preferred choice in modern data warehousing. Why? The performance gap has narrowed, and star schemas are faster, simpler, and easier for business users. For your project: use star schema unless you can justify otherwise.
