01

What Is a Data Warehouse?

📖 Official Definition — Memorise This

A data warehouse is a subject-oriented, integrated, time-variant, and non-volatile collection of data in support of management's decision-making process. (Inmon's definition)

In plain English: a data warehouse is a large, specially structured database that collects data from many sources and organises it so that managers and analysts can ask complex questions, find patterns, and make better decisions.

💬 Lecturer's Key Message

"For this unit, from week one until the final exam — assume you are a manager, not a developer. You are using the data warehouse to support decision making, not to record daily transactions."

Data Warehouse vs. Relational Database — The Big Picture

Relational Database
  • Supports: DELETE, INSERT, UPDATE, SELECT
  • One department or organisation
  • Simple, fast queries
  • Shows what is (facts only)
  • Passive — can't forecast or find trends
  • Current data only
Data Warehouse
  • Mainly supports: SELECT (complex queries)
  • Multiple departments / whole organisation
  • Complex analytical queries
  • Used for decision making & forecasting
  • Active — finds patterns and trends
  • Historical data (5–10 years)
02

Relational Database Recap

Before building a data warehouse, you need to understand what a relational database is — because operational databases are the primary source of data for a warehouse.

Key Concepts

  • Table: A 2D structure with rows and columns. Each table has a unique name.
  • Column (Attribute): Represents a property of an entity (e.g., Name, Age, Score).
  • Row (Record/Tuple): A single entry in the table — one real-world instance.
  • Primary Key (PK): A column (or combination) that uniquely identifies each row in a table.
  • Foreign Key (FK): A column that references the PK of another table — used to link tables together.

SQL Operations in a Relational DB

The main operations are join (combining tables), selection (filtering rows), and projection (choosing columns). The goal is fast retrieval, accurate updates, and data integrity.

Transactional Database

A transactional database stores records of individual events. For example: "Customer X bought item Y at time Z." This is called OLTP — Online Transaction Processing. Think: banking, supermarket checkouts, student enrolments.

🎯 Exam Point — Passive Nature of RDBMS

Relational databases are passive — they show you what happened, but can't tell you why or predict the future. That limitation is the core reason we need data warehouses.

03

The 4 Core Properties of a Data Warehouse

These four properties define what a data warehouse fundamentally IS. Every exam question about "what is a data warehouse" can be answered with these four words.

PropertyWhat it meansKey detail / exam point
Subject-Oriented Organised around major business subjects: Customer, Product, Sales, Time, Supplier. NOT focused on daily operations or transaction processing. Excludes data irrelevant to decision-making. A manager cares about Sales trends, not individual SQL updates.
Integrated Built by combining multiple, heterogeneous data sources (relational DBs, flat files, CSVs, online records). Data must be cleaned and standardised: "UWA" vs "Uni. of WA" vs "University of Western Australia" = same thing. Data integration = ~80% of project effort!
Time-Variant Keeps historical data — typically 5–10 years. Every key structure contains a time element. Operational DB = current values only. DW = historical perspective. "What were sales in Q3 2019?" is a DW question.
Non-Volatile Data in the DW is NOT updated or deleted during normal use. Only two operations: initial loading and access/reading. No transaction processing, recovery, or concurrency mechanisms needed. Once loaded, data stays. Exception: slowly changing dimensions (covered in Lecture 4).
🎯 Common Exam Question

"List and explain the four main properties of a data warehouse with an example for each." — Answer using the table above. A common trap is confusing integrated (multiple sources combined) with non-volatile (data not changed). They are different!

Why Integration Is Hard — The 80% Problem

  • Poor metadata: Without a data dictionary, you don't know what columns mean.
  • Data quality issues: Missing values, multiple spellings ("UWA" vs "University of WA"), inconsistent formats.
  • Semantic inconsistency: "HD" at UWA = 80+, but at another uni = 85+. Same label, different meaning.
04

OLTP vs. OLAP

📖 Definitions — Both Required

OLTP (Online Transaction Processing) = the major task of traditional relational databases. Records day-to-day operations (purchasing, banking, registration). Fast, small queries, frequent updates.

OLAP (Online Analytical Processing) = the major task of data warehouse systems. Used for data analysis and decision-making. Slow, complex queries on large amounts of data. Infrequent updates.

FeatureOLTPOLAP
PurposeRecord transactions / daily operationsAnalysis & decision making
UsersDatabase administrators, clerks, customersManagers, analysts, decision makers
Query typeShort, simple (a few records)Long, complex (millions of records)
Update frequencyVery frequent (constant)Infrequent (batch loading)
Data currencyMust be up-to-date alwaysHistorical snapshots are fine
SchemaNormalised (many tables, low redundancy)De-normalised (fewer tables, redundancy OK)
PrioritySpeed and consistencyQuery depth and analytical power
ExamplesUpdate account balance, enrol in course, add to cartTotal sales per department per month, top-selling books
🎯 Exam Trap — Don't Mix These Up

OLTP uses normalised schemas (many separate tables, low redundancy, high integrity). OLAP uses de-normalised schemas (fewer joins, redundancy acceptable for query speed). If you're asked "which schema does a DW use?" — de-normalised is correct for query performance. OLTP normalises; OLAP denormalises.

Example OLAP Queries (Complex — DW Territory)

  • Show a list of all items sold in the last quarter
  • Show total sales of the last month, grouped by branch
  • Which sales person had the highest amount of sales?
  • How many sales transactions occurred in September?

None of these can be easily answered by a simple OLTP system — they require aggregation, grouping, and often multiple source tables.

05

Why OLTP and OLAP Cannot Share a System

The slides give three clear reasons. These are likely exam short-answer material.

Reason 1: Different Performance Requirements

OLTP needs fast response times (under 1 second). OLAP queries consume massive CPU and disk resources. If both run together, an OLAP query can lock a table and block all OLTP transactions. Result: unhappy users, a crashed system.

Example: An OLAP query computing "sum of all sales" acquires a lock on the sales table. A new sales transaction is then blocked until the lock is released.

Reason 2: Different Data Modelling Requirements

OLTP uses normalised schemas — many tables, complex relationships, minimal redundancy. OLAP needs de-normalised schemas — fewer tables, simpler structure, easier for non-technical users to query. A schema designed for OLTP is slow and confusing for OLAP.

Reason 3: Analysis Needs Data from Many Sources

OLTP systems are built for one specific process (e.g., online ordering). OLAP integrates data from sales, inventory, HR, and purchasing — across departments and even different organisations. OLTP schemas are built around one department's needs, not enterprise-wide analysis.

✅ The Solution

Build a separate data warehouse that copies data from OLTP systems via ETL (Extract, Transform, Load). OLTP continues running fast; the DW handles analytical queries on a separate, optimised copy of the data. Both systems operate at peak performance independently.

Additional Reasons for Separation (from Slide 37)

  • Missing data: Decision support requires historical data; operational DBs don't keep it.
  • Data consolidation: DW requires aggregation and summarisation across heterogeneous sources.
  • Data quality: Different sources use inconsistent representations, codes, and formats that must be reconciled before analysis.
06

The Three-Tier Data Warehouse Architecture

The DW architecture has three layers. You need to know what each does and what sits in each tier.

TierNameWhat it containsPurpose
Bottom (Tier 1) Data Storage / Warehouse Server The actual data warehouse database (fact + dimension tables). Also: metadata repository, data marts. Stores all cleaned, integrated historical data. Source: operational databases + external sources, brought in via ETL.
Middle (Tier 2) OLAP Server The data cube (multi-dimensional data structure). Supports complex OLAP queries: roll-up, drill-down, slice, dice, pivot. Can be ROLAP (relational) or MOLAP (multi-dimensional).
Top (Tier 3) Front-End / Client Tools Reporting tools, dashboards, data mining tools. What the manager sees. Examples: Power BI, Tableau, Excel. Also includes ad-hoc query tools and decision support applications.
🎯 Exam Point — Where Does Metadata Sit?

The metadata repository sits in the bottom tier (data storage layer). It is fundamental — everything else depends on it. Metadata must be understood before data can be loaded. This is why it's at the foundation. (Full metadata coverage in Lecture 6.)

What Is a Data Mart?

A data mart is a smaller data warehouse designed for a single department (e.g., Sales, Finance). You can think of it as a subset of the enterprise data warehouse. The lecturer said: "A data mart is for one department; a data warehouse is for the whole company." (Full coverage in Lecture 6.)

What Is the OLAP Cube?

The cube is a multi-dimensional data structure in the middle tier. Instead of a flat table, data is organised across multiple dimensions (e.g., Product × Location × Time). This allows fast, flexible queries like "How many phones were sold in Chicago in Q1?" without running expensive SQL joins. Full coverage in Lecture 3.

07

ETL — Extract, Transform, Load

📖 Definition

ETL is the process of moving data from operational source systems into the data warehouse. It has three stages: Extract (pull data from sources), Transform (clean, standardise, integrate), Load (insert into the DW tables).

The lecturer emphasised: "The ETL part contributes the most marks in Project 1." It's also the most time-consuming part of any real DW project — roughly 80% of total effort.

Why Transformation Is Necessary

  • Inconsistent formats: Age stored as "31" in one system, "31 years" in another
  • Missing values: How do you handle NULLs? Drop them? Fill with mean/median?
  • Different naming conventions: "customer_id" vs "cust_num" vs "ClientID"
  • Semantic differences: "HD" means different things at different universities
  • Duplicates: The same customer appearing twice from two different source systems
💡 Tools Used in This Unit

Project 1 uses PostgreSQL (relational DB) and Databricks (ETL + DW). Visualisation: Power BI (Windows) or Tableau (Mac/Windows). You can also use Python, SQL, or R for ETL — any tool is acceptable as long as the result is justified.

Summary

Lecture 01 — Key Takeaways

  • A data warehouse = subject-oriented, integrated, time-variant, non-volatile collection of data for decision support.
  • Relational DBs are for OLTP (transactions); DWs are for OLAP (analysis). They need to be separate systems.
  • 3 reasons OLTP and OLAP don't mix: performance conflict, different data modelling needs, analysis requires data from many sources.
  • The 3-tier architecture: data storage (bottom) → OLAP server (middle) → front-end tools (top).
  • ETL brings data from source systems into the warehouse — it's ~80% of the real-world effort.
  • Assume you are a manager, not a developer, when answering exam questions.

Exam-Style Practice Questions

Define a data warehouse using Inmon's four properties. Explain each property with one example.

Hint: Subject-oriented (organised around Customer/Sales/etc.), Integrated (multiple sources combined), Time-Variant (5–10 years of history), Non-Volatile (load + read only).

What is the difference between OLTP and OLAP? Give two examples of queries for each.

Hint: OLTP = fast, small, transactional. OLAP = slow, large, analytical. Examples: OLTP: "update account balance"; OLAP: "total sales by branch for last quarter".

Explain three reasons why a data warehouse must be separate from an operational OLTP database.

Hint: Performance conflict (locking), different data modelling (normalised vs de-normalised), analysis needs data from many sources (cross-department).

Describe the three-tier architecture of a data warehouse. What does each tier contain?

Hint: Bottom = data storage + metadata repository + data marts. Middle = OLAP server (cube). Top = front-end tools (Power BI, Tableau, dashboards).

Why is data integration described as "the hardest part" of building a data warehouse? Give two specific challenges.

Hint: Poor metadata (don't know what columns mean), data quality (inconsistent spellings, missing values), semantic inconsistency (same label, different meaning at different universities).

A retail company runs its point-of-sale system (recording every sale in real time) on the same server as its business analytics system (generating monthly trend reports). What problem could arise and how would you fix it?

Hint: OLAP queries lock tables → OLTP transactions are blocked → angry users. Fix: separate into two systems with a DW fed by ETL from the OLTP.
