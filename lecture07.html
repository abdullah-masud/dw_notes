What Is a Graph?

📖 Definition

A graph is a collection of nodes (vertices) and edges (relationships). Nodes represent entities; edges represent connections between them.

Think of a road map — cities are nodes, roads are edges. Or a social network — people are nodes, friendships are edges. Any scenario involving interconnected entities can be modelled as a graph.

A graph is just nodes connected by edges. Roads between cities, friendships between people, citations between papers — all are graphs.

Real-world graph examples

  • Google Knowledge Graph: Connects entities (people, places, facts) to surface contextually relevant search results.
  • Social networks (X/Twitter, Facebook): Users = nodes; FOLLOWS, FRIEND_OF = edges.
  • Fraud detection: Traces suspicious patterns through connected transactions.
  • Connected Papers: Academic papers as nodes, citations as edges — shows related research.
02

Limitations of RDBMS & NoSQL

Problems with RDBMS for connected data

  • Slow multi-table joins: Answering "what did Alice order and what products were in those orders?" requires joining User → Order → LineItem → Product — 4 tables. Each join scans entire tables. The deeper the connection, the slower it gets.
  • Inflexible schema: Adding a new column forces ALTER TABLE and fills NULL for every existing row. Schema changes break applications and require planning.
  • Relationships are second-class: A foreign key is just a meaningless number. You know Employee 3 relates to Order 5678, but you don't know what that means without documentation.
  • Recursive queries are painful: Representing a management hierarchy (employee → manager → director → CEO) requires 15+ lines of complex recursive SQL. In a graph DB it's a few lines of Cypher.

🎯 Key Quote from Lecturer

"For the relational database, we focus on data storage. For the graph database, we focus on the relationship." RDBMS was designed to replace paper filing cabinets — it was never built to traverse deeply connected networks.

Problems with NoSQL

  • One-directional traversal: Easy to go Alice → orders, hard to go backwards from an order to Alice.
  • No referential integrity: No enforcement that updating one value updates all dependent references.
  • Poor physical locality: System doesn't know where related documents are in memory — expensive lookups.
💡 The Bottom Line

RDBMS: great at storing structured data, bad at traversing relationships. NoSQL: great for unstructured data, bad at maintaining connections. Graph DB: designed specifically so relationships are the primary concern — first-class citizens.

03

Graph Database vs. Graph Compute Engine

📖 The Analogy

Graph Database ≈ OLTP. Real-time, CRUD operations, daily use. Example: Neo4j.
Graph Compute Engine ≈ OLAP. Batch processing, global algorithms, not real-time.

Graph Database (OLTP-like)
  • Stores data persistently as a graph
  • Supports Create, Read, Update, Delete (CRUD)
  • Designed for real-time queries on connected data
  • Example: Neo4j (most popular)
  • Used in: social networks, fraud detection, recommendations
  • This unit's focus
Graph Compute Engine (OLAP-like)
  • Runs global graph algorithms on large datasets
  • Optimised for batch processing — not real-time
  • Data is extracted, transformed, then loaded
  • In-memory processing for speed
  • Examples: PageRank, community detection, shortest path across entire graph
  • Not this unit's focus

Native vs. Non-Native Graph Storage

  • Native (e.g. Neo4j): Data stored physically as a graph in memory. Each node has a direct pointer to its neighbours. Traversal follows pointers — no index scanning needed. This is called index-free adjacency.
  • Non-native: Uses traditional storage (relational/key-value) underneath, presents results in graph form. Slower traversal.
04

Why Graph Databases? — 3 Reasons

Reason 1 — Performance

In a graph DB, query time is proportional to the size of the subgraph traversed, not the size of the entire graph. You start at a node and follow its direct edges — the rest of the graph is untouched.

Depth RDBMS Neo4j
2 hops (friends of friends) ~2 seconds ~2 milliseconds
3 hops ~30 seconds ~4 milliseconds
4 hops > 10 minutes ~8 milliseconds
5 hops Infeasible ~1 second

Real experiment from the slides: "finding friends of friends" in RDBMS vs Neo4j. RDBMS degrades exponentially; Neo4j stays nearly constant per hop because it only touches the relevant subgraph.

🎯 Why Graph Is Faster — Index-Free Adjacency

Each node in Neo4j has a direct memory pointer to its neighbours. There is no need to look up a global index. In RDBMS, every join requires an index scan across the full table. The deeper you traverse, the worse RDBMS gets. Graph DB performance stays roughly constant per hop.

Reason 2 — Flexibility (Naturally Additive)

Graphs are naturally additive: you can add new labels, new relationship types, and new properties to specific nodes without breaking any existing queries and without a schema migration.

  • RDBMS: new relationship type → ALTER TABLE → NULL for all existing rows → potential query rewrites
  • Graph DB: add a WORKS_IN relationship to some nodes → nothing else changes → existing queries return identical results
💬 Lecturer

"If I want to add Helen works in Company A, just add the node for company and a WORKS_IN relationship. You don't need to change the schema because for graph database, we don't have a schema."

Reason 3 — Agility (Schema-Free)

  • No schema to define upfront — model as you go
  • No NULL values — if a property doesn't apply to a node, it simply isn't stored
  • No ACID constraints or normalisation rules (1NF, 2NF, 3NF) required
  • Natural language maps directly to the graph model
  • Schema-free means you don't need to plan everything in advance — you can extend ad hoc

Rules for Nodes

  • Can have zero or more labels — Harry is both a User AND Admin simultaneously
  • Can have zero or more properties — key-value pairs (name: "Alice", age: 28, email: "...")
  • If a property doesn't apply, it's simply absent — no NULLs ever stored

Rules for Relationships

  • Must be named: FOLLOWS, FRIEND_OF, SOLD, CONTAINS, REPORTS_TO
  • Must be directed: always have a start node → end node (shown with arrows)
  • Must have both a start AND end node — cannot connect to nothing. Self-connections (node → itself) are allowed.
  • Can also have properties: e.g. CONTAINS relationship can have unitPrice, quantity, discount
  • Relationships are first-class citizens — they are the whole point of a graph DB
🎯 Exam Point — Relationships Are First-Class

Lecturer: "In the graph database, we focus on the relationship. The edge is the first-order citizen." In RDBMS, a foreign key is just a number. In graph DB, the relationship is explicitly named (SOLD, FRIEND_OF) — you immediately understand what it means.

Two Pitfalls to Avoid (from Slides)

Pitfall 1 — Model to facilitate your queries (don't only capture WHO)

❌ Bad Model

Users directly connected by EMAILED:

(Bob) -[EMAILED]→ (Charlie)
(Bob) -[BCC]→ (Davina)

  • Cannot answer: "What was in the email?"
  • Cannot answer: "Who else received it?"
  • Only captures WHO, not WHAT

✅ Good Model

Separate Email node added:

(Bob) -[SENT]→ (Email)
(Email) -[TO]→ (Charlie)
(Email) -[BCC]→ (Davina)

  • Can query email content
  • Can find all recipients (TO, CC, BCC)
  • Captures WHO and WHAT

Pitfall 2 — Think ahead so the graph can evolve

❌ Bad Model

Only one type of relationship between emails:

  • Alice -[REPLIED_TO]→ Email 1234 -[TO]→ Charlie
  • Cannot distinguish: did Alice send, reply, or forward?
  • Cannot track reply chains

✅ Good Model

Separate email nodes per message:

  • Alice -[SENT]→ Email11 -[FORWARD_OF]→ Email12
  • Can distinguish sent, replied, forwarded
  • Can trace full thread history
🎯 Modelling Rule

From the slides: "Make sure your graph model facilitates the types of queries you are interested in." Before finalising, check every business query: can this model answer it? If not, add the missing nodes or relationships.

🎯 Exam Point — Identifying Join Tables

A join (relationship) table in RDBMS has exactly 2 foreign keys as its composite primary key and represents a many-to-many relationship. In graph DB: the join table disappears — it becomes a named relationship. Its non-key columns become relationship properties.

Special relationship types

  • One-to-many: Direct relationship from one node to many (one Supplier SUPPLIES many Products)
  • Many-to-many: Named relationship with properties — no join table needed in graph DB
  • Self-referential: Relationship from a node to another node of the same label (Employee REPORTS_TO Employee)

A student asked "seems like graph DB is just better — any downsides?" Lecturer: "The graph database contributes very small to industry. Most still use relational database because retrieval is easy — it's a table. Another issue is the lack of standard — everyone has their own standards."

No NULLs — A Key Structural Difference

In RDBMS, adding a "favourite_product" column to Users forces NULL for every user without a favourite product. In graph DB, you simply don't add that property to those nodes. Non-existing values are just absent — the property doesn't exist on that node at all. This makes the graph naturally sparse and avoids NULL pollution.

Short Answer Questions

1. What is a graph database?

A graph database is a database management system that uses a graph structure to store, map, and query relationships between data. Instead of tables and rows, it stores data as nodes (entities) and relationships (connections between entities), each of which can have properties (key-value attributes).

Key characteristics:

  • Built for OLTP — supports Create, Read, Update, Delete (CRUD) in real time
  • Optimised for traversing relationships — query time proportional to subgraph, not full dataset
  • Schema-free — no fixed table structure; can evolve without migration
  • No NULL values — absent properties are simply not stored
  • Best example: Neo4j (the most widely used graph database)
One-liner: A graph database stores data as nodes and relationships rather than tables and rows, making it ideal for highly connected data.
2. Define node, relationship, label, and property.
Term Definition Example
Node The fundamental unit of a graph — represents a real-world entity or object. Equivalent to a row in a relational table. A User named "Alice", an Order with ID 1234, a Product called "Laptop"
Relationship A named, directed connection between two nodes. Always has a start node and an end node. Can have properties. Equivalent to a foreign key — but explicitly named and meaningful. FOLLOWS (User → User), SOLD (Employee → Order), CONTAINS (Order → Product)
Label A type tag applied to a node to classify it. One node can have multiple labels simultaneously. A node can be labelled "User" AND "Admin" at the same time
Property A key-value attribute stored on a node OR a relationship. Optional — if a property doesn't apply, it is simply not stored (no NULL). name:"Alice", age:28 on a node; quantity:3, unitPrice:9.99 on a CONTAINS relationship
Memory hook: Nodes = things. Labels = what type of thing. Properties = details about the thing. Relationships = how things connect.
3. Why are recursive queries easier in graph databases?

In a relational database, recursive queries (e.g. "find all employees who report to this manager, and who reports to them, and so on") require complex SQL using Common Table Expressions (CTEs) or recursive WITH clauses — often 15+ lines of code, and still slow because each level requires joining the entire table against itself.

In a graph database (Neo4j/Cypher), the same query is a simple traversal: follow the REPORTS_TO relationship as many hops as needed. The query is just a few lines, and execution only touches the relevant nodes — not the full dataset.

RDBMS — Recursive Query

WITH RECURSIVE chain AS (
  SELECT emp_id, manager_id
  FROM employees WHERE emp_id = 5
  UNION ALL
  SELECT e.emp_id, e.manager_id
  FROM employees e
  JOIN chain c ON e.emp_id = c.manager_id
)
SELECT * FROM chain;

Slow for deep hierarchies. Joins entire table at each level.

Neo4j Cypher — Same Query

MATCH (e:Employee
  {emp_id: 5})
<-[:REPORTS_TO*]-
(subordinate)
RETURN subordinate

Fast. Only traverses the relevant subgraph. Depth specified with * (any depth).

Reason: Graph DB stores relationships as direct memory pointers — traversal follows pointers without scanning any table. RDBMS must join the entire table at every level of recursion.
4. Why are graph databases schema-free?

Graph databases are schema-free because they do not require a predefined structure (schema) to store data. In a relational database, you must define all tables, columns, and data types before inserting any data, and changing the schema is expensive (ALTER TABLE, data migration, NULL-filling).

In a graph database:

  • New node types (labels) can be added at any time without affecting existing nodes
  • New relationship types can be added between specific nodes without changing anything else
  • New properties can be added to individual nodes without affecting others — no NULLs for nodes that don't have that property
  • Existing queries continue working unchanged when new data is added

This makes graphs naturally additive — you build the model incrementally as your understanding of the domain grows, rather than planning everything upfront.

Schema-free = add new node types, relationship types, and properties at any time without breaking anything or migrating data.
5. How are NULL values handled in graph databases?

Graph databases have no NULL values. If a property does not apply to a node or relationship, it is simply not stored — the property is absent. There is no concept of storing a placeholder like NULL.

Comparison:

  • RDBMS: Adding a new column (e.g. "twitter_handle") means every existing row gets NULL for that column, even if 95% of users don't have Twitter. Storage is wasted and queries must handle NULLs.
  • Graph DB: Only add the "twitter_handle" property to the specific nodes where it exists. Other nodes simply don't have that property at all — no NULL stored, no storage waste, no NULL checks needed.
Rule: In graph DB, missing = absent. A property that doesn't exist on a node is simply not there — never NULL.
6. What is a property graph model?

The property graph model is the most common data model used in graph databases (including Neo4j). It defines exactly what a graph database can store:

Component Definition Key Rules
Nodes Represent entities (people, products, places, events) Can have 0+ labels, 0+ properties
Labels Type tags that classify nodes (like a table name) One node can have multiple labels
Properties Key-value pairs storing attributes on nodes or relationships Optional. If absent, simply not stored (no NULL)
Relationships Named, directed connections between nodes Must be named, directed, have start AND end node. Can have properties.

Together, these four components allow the graph to represent any real-world domain in a way that closely mirrors natural language: "Alice FOLLOWS Bob" maps directly to: node(Alice) -[:FOLLOWS]→ node(Bob).

Property graph model = nodes + relationships + labels + properties. Everything in Neo4j is built from these four pieces.

2. Explain why graph databases perform better for highly connected data.

Graph databases outperform relational databases on connected data because of a property called index-free adjacency. Here is the detailed explanation:

How RDBMS handles connected queries

When you ask "find all friends of Alice's friends," the RDBMS must:

  1. Scan the Users table to find Alice's row
  2. Join the Friendships table to find Alice's friends (scanning the full table)
  3. Join the Friendships table again to find those friends' friends (scanning the full table again)
  4. Each additional hop doubles the work — scanning grows exponentially with depth

How Graph DB handles connected queries

In Neo4j, each node has a direct pointer in memory to every node it is connected to. When you ask the same question:

  1. Locate Alice's node (one lookup)
  2. Follow her FRIEND_OF pointers directly to her friends' nodes (no table scan)
  3. Follow those nodes' FRIEND_OF pointers directly to friends of friends (no table scan)
  4. Each hop is just pointer-following — cost stays constant per hop regardless of total graph size

The performance difference

Depth RDBMS Neo4j
2 hops (friends of friends) ~2 seconds ~2 milliseconds
3 hops ~30 seconds ~4 milliseconds
4 hops > 10 minutes ~8 milliseconds
5 hops Infeasible ~1 second

The key insight: graph DB query time is proportional to the size of the subgraph traversed, NOT the total size of the graph. RDBMS query time grows with the number of rows in the full table at every hop.

Index-free adjacency = each node has direct memory pointers to neighbours. No table scanning needed. Cost per hop stays constant regardless of graph size.
3. Explain the conversion from relational database to graph database.

Converting an RDBMS to a graph database follows three simple transformation rules:

Row → Node  |  Table Name → Label  |  Join/FK → Relationship

Rule 1: Each Row becomes a Node

Every individual record (row) in a relational table becomes one node in the graph. The data from that row's columns becomes properties on the node.

  • Row: Alice | age:28 | email:alice@example.com → Node with properties: name:"Alice", age:28, email:"alice@example.com"

Rule 2: Table Name becomes a Label

The name of the relational table becomes the label applied to all nodes created from that table's rows. Labels classify nodes by type.

  • Table "Products" → every node created from Products rows gets the label Product
  • Table "Employees" → every node gets the label Employee

Rule 3: Foreign Keys and Join Tables become Relationships

This is the most important rule. There are three sub-cases:

  • Simple FK (one-to-many): Becomes a direct relationship. Employee.OrderID FK → SOLD relationship (Employee → Order)
  • Join table (many-to-many): The join table disappears entirely. It becomes a named relationship between the two entities. The join table's non-key columns become properties on that relationship. OrderDetails(OrderID, ProductID, quantity, unitPrice) → CONTAINS relationship with properties: quantity, unitPrice
  • Self-referential FK: Becomes a relationship from a node to another node of the same label. Employee.ManagerID → REPORTS_TO relationship (Employee → Employee)
The hardest part is identifying join tables (those with exactly 2 FKs as their composite PK) — these disappear and become named relationships.
4. Explain Neo4j ETL rules with examples.

The Neo4j ETL (Extract, Transform, Load) process converts relational data into a graph. The ETL tool applies the standard conversion rules automatically, but you need to understand what it does:

ETL Rule What Happens Northwind Example
Row → Node Each table row becomes one graph node Row in Orders table → one Order node with properties: orderID, orderDate, shipCity
Table name → Label The table name becomes the node's label (type) "Suppliers" table → every node gets label Supplier
Column → Property Each column value in the row becomes a property on the node productName, unitPrice, unitsInStock → properties on Product node
FK → Relationship A foreign key creates a named relationship between nodes Orders.EmployeeID FK → SOLD relationship (Employee -[SOLD]→ Order)
Join table → Relationship with properties Many-to-many join table becomes a relationship. Non-key columns → relationship properties. OrderDetails(OrderID FK, ProductID FK, quantity, unitPrice, discount) → CONTAINS relationship with properties: quantity, unitPrice, discount
Self-join → Self-relationship A foreign key pointing to the same table creates a self-referential relationship Employees.ReportsTo FK → REPORTS_TO (Employee -[REPORTS_TO]→ Employee)

Full Northwind Conversion Result

  • Supplier -[SUPPLIES]→ Product (Supplier.ProductID FK)
  • Product -[PART_OF]→ Category (Product.CategoryID FK)
  • Employee -[SOLD]→ Order (Orders.EmployeeID FK)
  • Employee -[REPORTS_TO]→ Employee (Employee.ReportsTo self-FK)
  • Order -[CONTAINS {qty, price, discount}]→ Product (OrderDetails join table)
Key ETL insight: Join tables (with 2 FK composite PKs) are the ones that become relationships — they disappear as nodes entirely.
5. Describe the property graph model in detail.

The property graph model is the data model used by Neo4j and most modern graph databases. It has exactly four components:

1. Nodes

The fundamental building blocks — represent real-world entities. Each node is a self-contained data record. In Neo4j, nodes are drawn as circles.

  • Can have zero or more labels
  • Can have zero or more properties (key-value pairs)
  • Can be connected to any number of other nodes via relationships

2. Labels

Type tags applied to nodes. Labels classify nodes by what they represent, similar to a table name in RDBMS.

  • A single node can have multiple labels simultaneously — e.g. Harry is both a "User" AND an "Admin"
  • Labels allow you to filter nodes by type in queries
  • Labels are not unique identifiers — they are category markers

3. Properties

Key-value pairs that store attribute data. Properties can exist on both nodes AND relationships.

  • Completely optional — a node with no properties is valid
  • No NULLs — if a property doesn't apply to a node, it is simply absent (not stored)
  • Values can be: strings, integers, floats, booleans, dates, or arrays
  • Example on node: name:"Alice", age:28, email:"alice@uwa.edu.au"
  • Example on relationship: quantity:3, unitPrice:9.99 on a CONTAINS relationship

4. Relationships

Named, directed connections between exactly two nodes (start node and end node). Relationships are the most important concept in a graph database — they are first-class citizens.

  • Must have a name (relationship type): FOLLOWS, SOLD, CONTAINS, REPORTS_TO
  • Must be directed: always has a clear start node → end node (shown with an arrow)
  • Must have both a start AND end node — a relationship cannot "float" unattached
  • Self-relationships are allowed: a node can have a relationship to itself (e.g. Employee REPORTS_TO Employee)
  • Can have properties: e.g. CONTAINS relationship has quantity, unitPrice, discount as properties
Summary: Nodes = entities. Labels = types of entities. Properties = attributes of entities/relationships. Relationships = how entities connect. All four together = the property graph model.

Scenario 2 — Employee reporting hierarchies: Why is a graph database better than RDBMS?

The Problem

An organisation wants to store a reporting hierarchy: Alice reports to Bob, Bob reports to Carol, Carol reports to the CEO. The hierarchy can be many levels deep and needs to be queried in multiple ways: "who does Alice report to?", "list everyone who reports to Bob directly or indirectly", "how many levels is Alice from the CEO?"

RDBMS Approach — The Problems

  • Self-join complexity: To find Bob's direct AND indirect reports, you need recursive SQL (WITH RECURSIVE CTE) — complex, verbose, and slow for deep hierarchies
  • Depth limitation: SQL queries must know the maximum depth in advance, or use complex loops
  • Performance: Each level of the hierarchy requires joining the entire Employees table against itself — gets exponentially slower for large organisations
  • Schema rigidity: If a new type of reporting relationship is added (e.g. "dotted-line" manager vs "solid-line" manager), the schema must change

Graph DB Approach — The Solution

  • Natural modelling: "Alice REPORTS_TO Bob" maps directly to a relationship — no join table needed
  • Simple traversal: Find all of Bob's reports at any depth with one Cypher line: MATCH (e)-[:REPORTS_TO*]->(bob) RETURN e
  • Multiple relationship types: Add DOTTED_LINE_REPORTS_TO relationship without changing anything else
  • Fast: Index-free adjacency means traversal follows pointers — doesn't scan the whole employee table at each hop

Graph Model

(Alice:Employee) -[:REPORTS_TO]→ (Bob:Employee)
(Bob:Employee) -[:REPORTS_TO]→ (Carol:Employee)
(Carol:Employee) -[:REPORTS_TO]→ (CEO:Employee)

Query — all of Bob's reports at any depth:
MATCH (emp)-[:REPORTS_TO*]->(bob {name:"Bob"}) RETURN emp

Employee -[REPORTS_TO]→ Employee (self-referential relationship). The * means "any number of hops" — Neo4j handles unlimited depth automatically.

Summary: Graph DB wins because: (1) natural modelling of hierarchies, (2) recursive traversal in one line of Cypher, (3) fast regardless of hierarchy depth, (4) schema-free — add new relationship types without migration.
Scenario 3 — Convert an e-commerce system (Orders, Products, Customers) from relational to graph DB.

Assumed Relational Schema

Table Columns Notes
Customers CustomerID (PK), Name, Email, Address Stores customer info
Products ProductID (PK), Name, Price, Category, Stock Stores product catalogue
Orders OrderID (PK), CustomerID (FK), OrderDate, Status CustomerID links to Customers
OrderItems OrderID (FK), ProductID (FK), Quantity, UnitPrice Many-to-many join table (Orders × Products)

Step 1: Apply the conversion rules

Rule 1 — Rows become Nodes:

  • Each row in Customers → one Customer node
  • Each row in Products → one Product node
  • Each row in Orders → one Order node
  • OrderItems has only 2 FK columns as PK → it's a join table → becomes a relationship (not a node)

Rule 2 — Table names become Labels:

  • Customers table → label: Customer
  • Products table → label: Product
  • Orders table → label: Order

Rule 3 — FKs and Join Tables become Relationships:

  • Orders.CustomerID FK → PLACED relationship: (Customer) -[:PLACED]→ (Order)
  • OrderItems join table (OrderID + ProductID composite PK) → CONTAINS relationship: (Order) -[:CONTAINS]→ (Product) with properties: quantity, unitPrice

Step 2: Final graph model

Graph Element Source Properties
Customer node Customers table rows customerID, name, email, address
Product node Products table rows productID, name, price, category, stock
Order node Orders table rows orderID, orderDate, status
PLACED relationship Orders.CustomerID FK (none needed)
CONTAINS relationship OrderItems join table quantity, unitPrice

E-commerce graph model. The OrderItems join table vanished — it became the CONTAINS relationship between Order and Product.

Benefits of the graph model over RDBMS here

  • Recommendation queries are trivial: "What products do customers who bought the same thing as Alice also buy?" — simple graph traversal vs complex multi-table SQL
  • No join overhead: Order to Products is a direct relationship — no join table to traverse
  • Extensible: Add REVIEWED, WISHLIST, RETURNED relationships without schema changes
  • No NULLs: If a product has no category, simply don't store a category property — no NULL needed
Key conversion: OrderItems (join table) → CONTAINS relationship with quantity and unitPrice as properties. The join table disappears as a node.
