01

The Starnet Model

📖 Definition

A starnet is a diagram used to visualise the multi-dimensional structure of a data warehouse for querying purposes. It shows dimensions as radial lines from a central point, with each line representing one dimension's concept hierarchy.

🎯 Exam Alert — Starnets Appear Every Year

The lecturer explicitly said: "Every year we have this question in the final exam — you design a starnet to answer the given query." You MUST know how to draw one correctly.

Required Components of a Starnet

  • Dimension names: Each line must be labelled with the dimension name (e.g., Location, Date, Product)
  • "All" node: Every dimension MUST have an "All" node. This represents the total/aggregate across that dimension.
  • Hierarchy levels: Each line must show the levels (e.g., Suburb → City → State → Country)

Footprints

📖 Definition

A footprint is a circle drawn through specific levels of each dimension to answer a particular query. It shows which granularity is being used for each dimension in that query.

Rules for footprints:

  • The footprint MUST form a complete circle — touching exactly one node per dimension
  • If a dimension is NOT used in the query, connect to its "All" node
  • Never leave any dimension unconnected — even unused dimensions must connect to "All"
  • A starnet contains ONLY dimension information — NEVER include measures or facts

Common Starnet Errors

⚠️ Common Mistakes — These Cost Marks

Error 1: Footprint is not a complete circle — missing a connection on one dimension. Error 2: Missing a dimension entirely (even unused dimensions must connect to "All"). Error 3: Including a measure or fact in the starnet (starnets = dimensions ONLY). Error 4: Incomplete cycle — not connecting through all hierarchy levels properly.

Starnet Design Example

Query: "How many X brand ice cream were sold in Australia in January 2026?"

Dimensions needed: Brand (item), Location, Date, Supermarket (if included). For this query: connect Brand → brand name level; Location → country level; Date → month level; Supermarket → All (not used in query).

💬 Lecturer on "How many dimensions in the starnet?"

"If query 1 asks for 3 dimensions, query 2 asks for 4, query 3 asks for 5 — you design ONE starnet with 5 dimensions because the starnet must be able to answer ALL queries."

02

Kimball's 4-Step Dimensional Modelling

This is a systematic approach to designing a data warehouse. The 4 steps are always done in order.

StepNameWhat you do
1Identify the ProcessChoose which business process to model (e.g., Ice Cream Sales, not all processes at once). Think about what business questions you need to answer.
2Determine the GrainDecide the most detailed level of one fact table row. Rule: go as fine-grained as possible (individual transaction if possible).
3Choose DimensionsIdentify candidate key from grain statement, then add other relevant dimensions functionally determined by that key.
4Identify MeasuresChoose numeric measures for the fact table. Classify each as Derived, Additive, Semi-Additive, or Non-Additive.
🎯 How Step 3 Works — Choosing Dimensions

Start with the grain statement. Example: "line item from cash register receipt" → candidate key = (Transaction ID, Product SKU). Then add more dimensions that are determined by the key: Store (where did the transaction happen?), Date (when?), Promotion (what promotion was active?). The candidate key gives you the minimum; business needs give you the rest.

03

Types of Numeric Measures (Step 4 Detail)

Measure TypeDefinitionExampleKey Rule
Derived FactCalculated from other columns within the same rowGross Profit = Revenue − Cost; Total = Price × UnitsPre-compute in ETL if frequently used (reduces latency, ensures consistency)
Additive FactCan be meaningfully summed across ALL dimensionsTotal Cost, Units Sold, RevenueThe most useful type — supports all OLAP operations without special handling
Semi-Additive FactCan be summed across SOME dimensions but NOT across timeBank account balance, Inventory levelSum by account or product is fine. Sum by date (Monday + Tuesday) is meaningless
Non-Additive FactCannot be summed across ANY dimensionUnit price, percentages, ratiosCannot be summed — use BI tools to calculate from other facts
💡 Why Pre-Compute Derived Facts?

If "Gross Profit" is queried 1000 times per day, recalculating Revenue − Cost every time wastes resources. Pre-computing it in ETL: reduces query latency, ensures consistent calculation (no different users applying different formulas), simplifies reports for business users.

04

The Data Cube

📖 Definition

A data cube is a multi-dimensional data structure that organises data around a central theme (e.g., Sales). It allows data to be viewed and analysed from multiple dimensions simultaneously. It is constructed from fact and dimension tables.

Key Concepts

  • Cells: Each intersection point in the cube holds a numeric measure (e.g., sales = 45). A cell is defined by a specific combination of dimension values.
  • Hypercube: When a data cube has more than 3 dimensions, it's called a hypercube.
  • Notation: (Jan, *, Phone, 150) means "in January, all locations, Phone product, sales = 150". * means "all values aggregated".

Why Use a Data Cube? (The Apple Analogy)

💬 Lecturer's Analogy

"For the cube solution — you already bought apples and put them in your fridge. You just take one out and eat it. For the database table solution — you want an apple, you go to the supermarket, buy it, come home, wash it, then eat it. The cube pre-prepares everything."

The benefit: pre-computed aggregations mean queries are answered instantly by reading from the cube rather than recalculating every time.

Reading Cube Notation

Format: (Date, Location, Item, Measure). An asterisk (*) means "aggregated across all values of that dimension."

  • (Jan, *, *, 2800) = Total sales across all cities and all items in January = 2800
  • (*, Perth, *, 1200) = Total sales in Perth across all months and items = 1200
  • (Jan, Perth, Phone, 45) = Phone sales in Perth in January = 45
05

Cuboids and the Lattice

📖 Definition

A cuboid is a subset of the full data cube — a "chunk" representing data aggregated at a specific combination of dimension levels. A lattice of cuboids is the complete set of all possible cuboids for a given set of dimensions.

Key Cuboid Concepts

  • Base cuboid (bottom): The most detailed — no aggregation. Contains all dimension combinations.
  • Apex cuboid (top): The most general — everything aggregated into a single total cell. Always labelled "All".
  • Each cuboid represents one possible GROUP BY combination.

How to Calculate the Number of Cuboids

🎯 Formula — Must Know for Exam

No hierarchies: If you have n dimensions with no levels, total cuboids = 2n

With hierarchies: If dimension i has Li levels, total cuboids = (L1+1) × (L2+1) × ... × (Ln+1)

💡 Worked Example

Date dimension: 4 levels (Day, Month, Quarter, Year). Location dimension: 3 levels (City, State, Country).
Cuboids = (4+1) × (3+1) = 5 × 4 = 20 cuboids.

Why +1? Because "All" (total aggregation) counts as an additional level at the top.

Drawing a Cuboid Lattice

The lattice diagram must include:

  • An "All" node at the top (apex)
  • All dimension combinations at the bottom (base cuboid)
  • All intermediate cuboids labelled with their dimension names
  • Arrows showing the direction of roll-up (bottom to top)

For ROLLUP(colour, month, state): the path is colour+month+state → colour+month → colour → All. The rightmost dimension is dropped first (state), then month, then colour.

06

SQL GROUP BY CUBE and ROLLUP

🎯 Exam Note — No SQL Writing Required

The lecturer confirmed: "In the final exam, you don't need to write any SQL query. You just need to explain something." But you must understand what CUBE and ROLLUP produce.

The Problem with Regular GROUP BY

A normal GROUP BY state, month query only gives you data for each (state, month) combination. It does NOT give you: subtotals per state, subtotals per month, or a grand total. Without CUBE or ROLLUP, you'd need multiple UNION ALL queries — one per combination.

GROUP BY CUBE

Computes ALL possible subtotals for all combinations of the grouped attributes. For CUBE(month, state) this produces: (month, state), (month, All), (All, state), (All, All) — 4 combinations = 2² = 2n.

NULL values in the result represent "All" for that dimension.

GROUP BY ROLLUP

Computes subtotals along ONE PATH through the lattice — following the order of the attributes listed. ROLLUP(A, B, C) produces: (A,B,C), (A,B), (A), () [grand total]. Order matters — it drops the last attribute first.

CUBE
  • Computes the ENTIRE lattice
  • All 2n combinations
  • n=2: gives 4 subtotal sets
  • More comprehensive
ROLLUP
  • Computes ONE PATH through lattice
  • Order of GROUP BY matters
  • Drops last attribute first
  • Fewer rows, less comprehensive

Summary

Lecture 03 — Key Takeaways

  • Starnet = diagram showing dimensions with hierarchy levels. Required: dimension names, "All" nodes, footprint as complete circle touching all dimensions.
  • Footprint answers a specific query — must touch one node per dimension (unused = "All").
  • Kimball's 4 steps: (1) Identify process, (2) Determine grain, (3) Choose dimensions, (4) Identify measures.
  • Data cube = multi-dimensional structure pre-computing aggregations for fast queries.
  • Cuboid = subset of the cube (a specific GROUP BY combination).
  • No hierarchies: cuboids = 2n. With hierarchies: ∏(Li+1).
  • CUBE = entire lattice (all combinations). ROLLUP = one path (drops rightmost first).

    Exam-Style Practice Questions

    A supermarket manager wants to analyse ice cream sales. The business queries are: (1) Which brand had highest sales in Claremont in 2025? (2) Total ice cream sales in Perth in Q4 2025? (3) Which supermarket sold most units in WA in December 2025? Draw a starnet and identify the dimensions and measures.

    Hint: Dimensions: Brand, Location (Suburb→City→State), Date (Month→Quarter→Year), Supermarket. Measure: units sold / sales.

    A data cube has 3 dimensions with no hierarchies: Product, Month, State. How many cuboids are in the lattice? List them all.

    Hint: 2³ = 8 cuboids: All; Product; Month; State; Product+Month; Product+State; Month+State; Product+Month+State.

    A data cube has 2 dimensions: Date (Day, Month, Quarter, Year = 4 levels) and Location (City, State, Country = 3 levels). How many cuboids are in the lattice?

    Hint: (4+1) × (3+1) = 5 × 4 = 20 cuboids.

    Explain the difference between GROUP BY CUBE and GROUP BY ROLLUP. When would you use each?

    Hint: CUBE = all possible subtotal combinations (entire lattice). ROLLUP = one hierarchical path, drops last attribute first. ROLLUP used for hierarchical summaries; CUBE used for full cross-dimensional analysis.

    A student claims that "inventory level" is an additive fact because you can add the inventory of different products together. Is this claim fully correct? Explain.

    Hint: Semi-additive. You CAN add across products (total inventory). You CANNOT add across time (Monday inventory + Tuesday inventory ≠ meaningful total — it double-counts).
