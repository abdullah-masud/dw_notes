What Is Metadata?

ðŸ“– Key Definition â€” Memorise This

Metadata = data about data. It describes, organises, and explains data â€” but it is NOT the data itself. It tells you what the data is, where it is, and how to use it.

Think of it this way: if a book is the data, then its title, author name, ISBN, table of contents, and index are its metadata. You can find and understand the book without reading every word â€” the metadata tells you what's inside.

"The dataset dictionary we provided for your project is metadata â€” it tells you what each column means, the data type, and the values. You already used metadata!"

Three things metadata does:

  • Leads us to detail â€” describes the data precisely
  • Helps organise & find data â€” acts like a search index
  • Helps understand data â€” explains meaning without opening the data itself

Metadata in a Relational Database

Relational databases store metadata in a structure called the data dictionary or system catalog. It holds: table names, column names, data types, constraints, and relationships. This is queried automatically by the DBMS engine.

Metadata in Data Warehouse Architecture

The DW has three layers (top = users/reports, middle = OLAP server, bottom = data storage). The metadata repository sits at the bottom layer, separate from but connected to everything.

ðŸŽ¯ Exam Question â€” Why at the Bottom?

The lecturer was asked this directly and gave a clear answer: "Metadata is fundamental for everything in the data warehouse. Before you load data, you need to know the tables, the column meanings, where data comes from. Everything depends on it." The metadata repository must exist before data can be loaded.

What metadata describes for Tables vs. Columns

Tables
  • Physical Name â€” actual DB name (e.g., dim_customer)
  • Logical Name â€” business-friendly label (e.g., "Customer Info Dimension")
  • Type â€” Fact or Dimension table
  • Role â€” Legacy, OLTP, or Stage
  • DBMS â€” which software (Oracle, SQL Server, etc.)

  • Location â€” which server/database

  • Physical & Logical Name

  • Datatype â€” VARCHAR, INTEGER, DATEâ€¦
  • Length â€” max characters/bytes
  • Default Value â€” if nothing is entered
  • Nullable/Required â€” can it be NULL?
  • Edit Rules â€” e.g., gender must be 'M' or 'F'

Physical name = what the database system uses (e.g., dim_customer). Logical name = what business users call it (e.g., "Customer Information Table"). The lecturer said: "You can't say to a manager 'go look at the dim_customer table'. You explain it in plain English."

The 7 Kinds of DW Metadata

Exam Important

The lecturer introduced all 7 in the transcript, explaining each with an example. Memorise all 7 names and what question each answers.

#TypeWhat question it answersSimple explanation
1 Data Definition & Mapping Metadata "What does this column mean?" & "Where did this data come from?" Lists every column in every table (DDS, ODS, NDS) with meanings, sample values, and source info. Uses table keys and column keys instead of names. Also called data lineage metadata.
2 Data Structure Metadata "How is the data organised?" Describes the structure of tables: column names, data types, relationships. Think of CREATE TABLE statements and ER diagrams.
3 Source System Metadata "What does the original data source look like?" Describes the data structure of external/internal source databases before ETL. Different from mapping â€” mapping says "where it came from"; this says "what the source looks like".
4 ETL Process Metadata "What happened during loading?" Like an ETL log file. Records: what transformations were applied, where data came from, how data was cleaned, and how it was loaded. The lecturer called it "the log file".
5 Data Quality Metadata "Is the data good?" Describes data quality rules (e.g., no missing values in PK columns), risk levels, and actions taken. Like ETL rules: do we allow NULLs? How do we handle them?
6 Audit Metadata "What happened in the warehouse, and when?" A record of all processes and activities: who accessed what, when was data updated, what queries ran. Like an access log.
7 Usage Metadata "How is the warehouse being used?" Event log of application usage: who uses which reports, when, how often. Helps optimise performance and understand business needs.

Deep Dive: Data Definition vs. Mapping Metadata

These two are closely related but different. The slides and lecturer clarified:

Data Definition Metadata

Describes every column in the DW â€” its meaning, description, and sample values. Uses numeric Table Keys and Column Keys (not actual names) to reference tables and columns.

Example: "Table Key 56, Column Key 12 = income in Australian dollars after tax"

Mapping (Lineage) Metadata

Tracks where each piece of data in the DW came from in the source system. The data lineage trail. Critical for error tracing â€” if a value is wrong, you can trace it back to the original source and identify whether the error is in the source or ETL.

Example: "DW column X came from source DB column Y, loaded at time Z"

ðŸ’¡ Why Do We Need Mapping Metadata? (Lecturer's Example)

"If something is wrong â€” if one data point is not correct â€” we need to go back and check: is it wrong in the original database, or was there a mistake in the ETL? Without mapping metadata, this takes a very long time. With it, you can trace immediately where it came from and how it was processed."

Metadata Repository

ðŸ“– Definition

A Metadata Repository is a central catalogue where all metadata is collectively stored, organised, and accessed. It's like a library database that doesn't contain the books, but stores everything about them.

Each Data Warehouse has one or more metadata repositories. The lecturer used the analogy of a library's database system â€” it stores book titles, authors, ISBN numbers, locations, etc., but not the actual book content.

What a Metadata Repository Holds:

  • Structure definition â€” what tables exist (fact tables, dimension tables, schemas)
  • Table structures â€” columns, data types, constraints, relationships
  • Data flow description â€” the ETL pipeline: extract â†’ transform â†’ load
  • Outcomes of processes â€” ETL logs (success/failure records)
  • Who accesses what & when â€” security and audit information
  • Aggregation info â€” pre-calculated summaries like monthly sales totals
ðŸŽ¯ Exam Short-Answer Setup

Know the difference: a metadata repository stores metadata (descriptions, structures, lineage), while the data warehouse stores the actual data (the rows and facts). They are separate but connected.

The 3 Categories of Metadata

Exam Important

The slides categorise all metadata into three types. Know each type, its characteristics, who uses it, and examples.

1. Business Metadata

For: Business people â€” managers, analysts, decision-makers

Uses logical (plain English) names, not physical database names

  • Data ownership information
  • Business definitions of fields
  • Changing policies & rules
  • Report/query definitions
  • High-level field descriptions
  • Info about cubes, aggregates, Data Marts

Example: "Revenue = total income minus costs minus GST"

2. Technical Metadata

For: DBAs, ETL developers, data engineers

Uses physical details â€” actual table names, types, storage locations

  • Primary & foreign keys
  • Indices
  • Data types & lengths
  • Physical storage location (server, database)
  • Data mapping & transformation info
  • System functions & query optimisation

Example: "Column gender is VARCHAR(2), stores 'M' or 'F'"

3. Operational Metadata

For: Data ops teams, system admins, ETL operators

Connects the repository to what's actually happening

  • Currency of data: is it active, archived, or purged?
  • Lineage of data: history of migrations & transformations
  • Source & target locations
  • File sizes, server info
  • Referenced at row level (not just table level)

Example: "Row 4521 was loaded on 12/03/2024 from CRM_DB after null-filling transformation"

ðŸŽ¯ Common Exam Trap â€” Operational vs. Technical

These two are easily confused. Remember: Technical metadata describes the structure of data (what type is it, how is it stored?). Operational is about what the data is doing; Technical is about what the data looks like.

Roles of Metadata in a Data Warehouse

Application 1: Finding Data

Data warehouses can hold terabytes of data. Searching the data directly would be impossibly slow. Metadata acts as a small, fast index â€” you query the metadata (column names, table names) to find the right data, then retrieve only that.

Application 2: Using Data

Metadata lets you understand the structure, terminology, and rules of the DW without looking at individual rows. A manager can understand what "monthly_revenue" means from its business metadata without running a query.

Functional Roles (from Slide 31)

  • Helps decision support systems locate warehouse contents
  • Supports data mapping during ETL â€” transforming data from operational to DW environment
  • Used by query tools â€” your SQL queries use table/column metadata
  • Used in extraction, cleansing, reporting, and transformation tools
  • Plays a role in loading functions â€” knowing which table/column to load into

Challenges of Metadata Management

âš ï¸ Lecturer Said â€” Not in Final Exam

"The challenge part â€” it's not in the final exam because it's the business part." You should still understand the concepts, but don't memorise them as deeply.

How Metadata is Managed (3 Ways)

  • People (Individual): Staff must be trained to understand and use metadata. If no one reads it, it's useless.
  • Process: Metadata management must be built into the DW lifecycle. E.g., when you create a new table, you document it immediately.
  • Tools: Metadata managers/repositories, ETL tools, BI reporting tools (like Power BI), data modelling tools.

The 3 Core Challenges

Challenge 1 â€” Disparate Information Sources

In large organisations, metadata comes from many systems. "Customer ID" in one department might be "Customer Number" in another. Ensuring consistent, unified metadata across all sources is very difficult.

Challenge 2 â€” Enforcing Business Rules

Metadata must follow business rules (e.g., is price inclusive or exclusive of GST?). Without rules, metadata becomes inconsistent and untrustworthy. Different departments may have different standards.

Challenge 3 â€” Effective Communication

There are no industry-wide accepted standards for metadata. Within a company, everyone must use the same standard. If metadata is too technical, business users won't use it. If it's out of date, it's dangerous for decision-making.

"The user manual is like metadata â€” it's there, but no one reads it. When something goes wrong, you go to ChatGPT instead of reading the manual. The challenge is getting people to actually use the metadata they already have."

Data Lake

ðŸ“– Definition

A Data Lake is a storage platform that stores any type of data â€” structured, semi-structured, unstructured, and binary â€” at any scale, in its raw format with no transformation or indexing required. Data is stored first, analysed later.

Key Characteristics:

  • All data types: Tables, images, videos, text documents â€” anything
  • One central repository: Everything in one place
  • Flexible analytics: Supports data science, machine learning, real-time analytics

Data Lake â‰  Data Warehouse

  • Flexible â€” supports ML, big data, real-time

What Is a Data Mart?

ðŸ“– Key Definition â€” Memorise This

A Data Mart is a subset of a data warehouse, focused on a specific business line or department (e.g., Sales, Marketing, Finance). It contains only the data relevant to that area, in a summarised, ready-to-query format.

"A data mart is a smaller, focused version of a data warehouse â€” built for one department, not the whole enterprise."

When Should You Use a Data Mart?

  • You need access control â€” only certain users should see certain data
  • A department needs faster query results without scanning the entire DW
  • A department needs data on different hardware/software platforms
  • Data needs to be structured differently to suit specific tools

Key Characteristics of a Data Mart

  • Single subject matter â€” dedicated to one business function
  • Consolidates & integrates relevant data from various sources
  • Built using dimensional model â€” typically uses star schema
  • Supports multidimensional analytics â€” OLAP-capable
  • Smaller than a DW â€” usually under 100 GB

Why Use a Data Mart? (Benefits)

  • Faster performance â€” less data to scan
  • Easier access â€” only relevant data is present
  • Lower cost than building a full enterprise DW (from $10,000+)
  • Faster to implement â€” 3â€“6 months vs. 1+ year for a full DW
  • Agile â€” smaller, easier to change if requirements shift
  • Granular access control â€” departments control their own data

DW Architecture with Data Mart (Slide 49)

The data flows: Source Systems â†’ ETL â†’ Data Warehouse â†’ Data Mart â†’ Business Users. Data marts sit between the DW and end users, providing a filtered, domain-specific view.

Inmon vs. Kimball: Two Approaches to DW Design

Exam Important

Bill Inmon and Ralph Kimball are the two founding fathers of data warehousing. They have fundamentally different philosophies. You must know both and be able to compare them.

Bill Inmon â€” Top-Down

Philosophy:

Build the enterprise data warehouse first, then create data marts from it. The DW is the single source of truth.

Process:

  1. Build centralised corporate data model
  2. Data Warehouse is the physical representation of this model
  3. Data Marts are created from the DW when needed

Key point:

DW is the source of all data marts. Data flows: Sources â†’ DW â†’ Data Marts.

Data integration:

Ensures data integrity & consistency across the entire organisation because everything comes from one place.

Ralph Kimball â€” Bottom-Up

Philosophy:

Build data marts first around key business processes. The DW is simply the collection of all data marts.

Process:

  1. Identify most important business processes
  2. Create Data Marts for each subject area
  3. The "DW" = the union of all data marts

Key point:

"The data warehouse is nothing more than the union of all Data Marts." â€” Kimball's famous quote.

Data integration:

Each data mart is built around business needs. Aggregated together they form the DW. Faster to deliver initial value.

Implementing a Data Mart

The slides give a clear 5-phase process. This is likely to appear as a short-answer question.

1
Designing

Gather business & technical requirements. Identify data sources. Select the appropriate data subset. Design the logical and physical structure (tables, indexes, star schema).

2
Constructing

Implement the physical database from the design. Create tables, indexes, views. No data yet â€” just the structure.

3
Populating

Map source data to target. Extract from source systems. Clean and transform (ETL). Load into the data mart. Create and store metadata.

4
Accessing

Query data, create reports and charts. Set up & maintain DB structures. Configure APIs and interfaces for end users.

5
Managing

Ongoing user access management. System optimisation & fine-tuning. Adding fresh data continuously. Planning recovery scenarios and ensuring availability.
