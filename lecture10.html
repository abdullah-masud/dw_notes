01

Why Do We Need Aggregation?

Sometimes you don't want to see individual records — you want to summarise them. For example:

  • "For each person, how many friends do they have?" — you want a count per person, not a list of every friend
  • "For each movie, list all actors who appeared in it" — you want a group per movie
  • "For each person named Jack, list their pets" — you want a list per Jack
💬 Lecturer's Explanation

"The keyword is 'for each something'. In SQL, whenever you say 'for each', you use GROUP BY. In Cypher, there is no GROUP BY keyword — but it works the same way. The non-aggregated column becomes the grouping key automatically."

How Cypher's Implicit Grouping Works

In SQL you write: SELECT person, COUNT(*) FROM friends GROUP BY person

In Cypher, you write: MATCH (p1)-->(p2) RETURN p1.name, COUNT(p2)

There is no GROUP BY keyword. Cypher is smart enough to know that p1.name (the non-aggregated column) is the grouping key, and COUNT(p2) is the aggregation. This is called implicit grouping.

📖 Implicit Grouping — The Core Rule

In Cypher's RETURN clause: any non-aggregated expression automatically becomes the GROUP BY key. Any aggregation function (COUNT, COLLECT, AVG, etc.) is applied per group. You never type GROUP BY.

02

COUNT — Counting Things

📖 What COUNT does

COUNT returns the number of results for each grouping key, instead of listing the items themselves. It's the most basic aggregation function.

COUNT(variable) — count non-null values

Count how many people each person follows
MATCH (p1:Person)-[:FOLLOWS]->(p2:Person) RETURN p1.name, COUNT(p2)
✅ In plain English: "For each person (p1), count how many people they follow (p2)."
p1.name is the grouping key (like GROUP BY). COUNT(p2) counts how many times p2 appears for each p1.
Result: Jack | 5, Freddy | 12, Jane | 7, Wanda | 16

COUNT(*) vs COUNT(column) — what's the difference?

COUNT(p.age)

Counts only rows where p.age is NOT null.

If 3 people have no age property → they are excluded from the count.

Use when you want to count items that actually have a specific value.

COUNT(*)

Counts ALL rows including those with NULL values.

If 3 people have no age → they are still counted.

Use when you just want the total number of rows/matches.

🎯 Exam Trap — COUNT never returns NULL

Most aggregation functions return NULL if all inputs are NULL. But COUNT always returns 0 (never NULL) even if there are no matching records. COLLECT always returns an empty list []. These are the two exceptions.

Without COUNT vs With COUNT — the difference

Without COUNT — returns individual results (each row)
MATCH (p1:Person)-[:FOLLOWS]->(p2:Person) RETURN p1.name, p2.name // Result: Jack | Alice, Jack | Bob, Jack | Carol, ...
With COUNT — returns aggregated result (one row per group)
MATCH (p1:Person)-[:FOLLOWS]->(p2:Person) RETURN p1.name, COUNT(p2) // Result: Jack | 3 (one row per person, not one per follow)
03

COLLECT — Building a List

📖 What COLLECT does

COLLECT groups all values for each grouping key into a single list. Instead of returning many rows (one per value), you get one row per group with all values bundled into a list.

Basic COLLECT example

For each person, collect a list of all their friends' names
MATCH (p1:Person)-[:KNOWS]->(p2:Person) RETURN p1.name, COLLECT(p2.name)
✅ In plain English: "For each person p1, collect all names of people they KNOW into a list."
Result: Jack | ["Alice", "Bob", "Carol"] — one row per person, with all friend names bundled in a list.
Without COLLECT: Jack | Alice, Jack | Bob, Jack | Carol — three separate rows.

COLLECT aggregates ages too

Collect ages into a list
MATCH (p1:Person)-[:KNOWS]->(p2:Person) RETURN p1.name, COLLECT(p2.age) // Result: Jack | [28, 35, 22] — list of ages of Jack's friends
💡 COUNT vs COLLECT

COUNT = how many? Returns a number.
COLLECT = list them all. Returns a list.
Both are group-by operations. COUNT(p2) → 3. COLLECT(p2.name) → ["Alice","Bob","Carol"].

04

Other Aggregation Functions

Function What it does Example NULL behaviour
COUNT(n) Count non-null values RETURN COUNT(p.age) Returns 0 (not NULL) if no matches
COUNT(*) Count all rows including NULLs RETURN COUNT(*) Always 0 minimum
COLLECT(x) Bundle all values into a list RETURN COLLECT(p.name) Returns [] (empty list) if no matches
AVG(n) Average of numeric values RETURN AVG(p.age) Returns NULL if all inputs are NULL
SUM(n) Total of numeric values RETURN SUM(p.salary) Returns 0 if no matches
MIN(n) Smallest value RETURN MIN(p.age) Returns NULL if all inputs are NULL
MAX(n) Largest value RETURN MAX(p.age) Returns NULL if all inputs are NULL
STDEV(n) Standard deviation RETURN STDEV(p.score) Returns NULL if all inputs are NULL
🎯 What Cypher Does NOT Have

Cypher only has these limited aggregation functions. It does NOT have: MEDIAN, MODE, VARIANCE, PERCENTILE (by default), STRING_AGG, or any window functions (RANK, SUM OVER, etc.). For these — use SQL. APOC adds some extras, but the exam won't test APOC aggregations.

Aggregation on all nodes (database statistics)

Count total nodes in the database
// Count ALL nodes MATCH (n) RETURN COUNT(n) // Count all relationships MATCH ()-[r]->() RETURN COUNT(r) // Average age of all people MATCH (p:Person) RETURN avg(p.age)
💡 MATCH (n) with no label or property filter matches ALL nodes. Used for checking database stats — "how many nodes do I have?"
05

ORDER BY, LIMIT, and SKIP

📖 Sorting and Pagination

These clauses let you sort and paginate results — just like SQL. ORDER BY sorts. LIMIT restricts how many rows are returned. SKIP skips the first N rows.

ORDER BY — sorting results

ORDER BY — sort results
// Sort by name ascending (default) MATCH (p:Person) RETURN p.name ORDER BY p.name // Sort by age descending (youngest last → oldest first) MATCH (p:Person) RETURN p.name, p.age ORDER BY p.age DESC
⚠️ NULLs and ORDER BY

If a property is NULL for some nodes (e.g. some people have no age), those NULL values appear first in ASC order and last in DESC order. Lecturer: "null will be the first one" when ordering ascending.

LIMIT — restrict results

LIMIT — return only the first N results
// Return only the 3 youngest people MATCH (p:Person) RETURN p.name, p.age ORDER BY p.age LIMIT 3

SKIP — skip the first N results

SKIP — skip first N results (pagination)
// Skip the 3 youngest, return the rest MATCH (p:Person) RETURN p.name, p.age ORDER BY p.age SKIP 3 // Page 2 of results (items 4-6) MATCH (p:Person) RETURN p.name ORDER BY p.name SKIP 3 LIMIT 3
💡 SKIP + LIMIT together = pagination. Skip the first page (3 items), return the next page (3 items).

Combining ORDER BY with aggregation (via WITH)

Count followers and sort by count descending
MATCH (p1:Person)-[:FOLLOWS]->(p2:Person) WITH p1.name AS person, COUNT(p2) AS followCount RETURN person, followCount ORDER BY followCount DESC LIMIT 5
✅ In plain English: "Count how many people each person follows. Return the top 5 with the most follows, sorted highest first."
06

UNWIND — Turning a List Back into Rows

📖 UNWIND = the opposite of COLLECT

COLLECT turns many rows into one list. UNWIND takes a list and turns it back into individual rows — one row per item in the list. Think of it as "exploding" a list into rows.

UNWIND — expand a list back into rows
// Collect names first, then unwind back to rows MATCH (p:Person) WITH COLLECT(p.name) AS names UNWIND names AS name RETURN name
✅ Step 1: COLLECT all names into one list: ["Alice","Bob","Carol"]. Step 2: UNWIND the list back into individual rows. Step 3: Return each name as a separate row. Result is the same as just MATCH (p:Person) RETURN p.name — this is mainly useful when you need to manipulate the list first.
UNWIND with a hardcoded list — useful for batch operations
// UNWIND can also expand a literal list UNWIND [1, 2, 3, 4, 5] AS number RETURN number // Returns 5 rows: 1, 2, 3, 4, 5
07

List Comprehension & Pattern Comprehension

📖 What is List Comprehension?

List comprehension is a compact way to create a new list from an existing list or pattern, with an optional filter. If you've used Python, it's the same idea: [x for x in list if condition].

Python analogy (from the slides)

Python — list comprehension (recall)
# Long way: fruits = ["apple", "banana", "cherry", "kiwi", "mango"] newlist = [x for x in fruits if "a" in x] # Result: ["apple", "banana", "mango"]

List Comprehension in Cypher

Cypher list comprehension syntax
// Syntax: [variable IN list WHERE condition | expression] // Example: from a list of numbers, keep only those > 2 and return squared RETURN [x IN [1,2,3,4,5] WHERE x > 2 | x * x] // Result: [9, 16, 25] // Filter a list: only names containing "an" RETURN [name IN ["Alice", "Andrea", "Bob"] WHERE name CONTAINS "an"]

Pattern Comprehension — list from a graph pattern

Pattern comprehension — collect specific values from matched patterns
// For each person, collect the names of all movies they acted in MATCH (p:Person) RETURN p.name, [(p)-[:ACTED_IN]->(m:Movie) | m.title] AS moviesActedIn
✅ In plain English: "For each person, return their name and a list of all movie titles they acted in."
The pattern (p)-[:ACTED_IN]->(m:Movie) is matched for each person, and m.title is collected into a list. This is more concise than using WITH + COLLECT.

What is APOC?

📖 Definition

APOC stands for Awesome Procedures on Cypher. It is a library of over 450 functions and procedures that extend Neo4j's built-in Cypher capabilities. Think of it like a Python library — it's not built into the language but you can install and import it.

💬 Lecturer's Analogy

"APOC is just like a Python library. You install it, import it, and then you have access to lots of extra functions. For the final exam, you don't need to memorise APOC queries. But you need to understand what APOC is and what it can do."

What APOC Provides

  • Data import/export: Faster and more flexible CSV/JSON import than plain LOAD CSV
  • Dynamic relationships: Create relationship types based on values in a CSV column
  • Text functions: String manipulation (capitalize, trim, split, join, etc.)
  • Path expansion: More control over traversal (filter by relationship type, label, direction, depth)
  • Virtual graphs: Create temporary nodes/relationships that exist only in memory (not stored in DB)
  • Graph algorithms: Connectivity, centrality, community detection

APOC Syntax — CALL and YIELD

APOC procedures are called using CALL and their results are accessed using YIELD:

APOC general syntax
CALL apoc.procedure.name(arg1, arg2, ...) YIELD resultColumn RETURN resultColumn
💡 CALL invokes the APOC procedure. YIELD "catches" the output columns from the procedure so you can use them in your RETURN clause.

APOC — Importing CSV Data

APOC offers apoc.import.csv which can import both nodes and relationships from CSV files in a single command — much simpler than writing separate LOAD CSV statements for each node type and each relationship type.

🎯 Project 2 Rule — When You Can Use APOC

Lecturer: "For Project 2 — you are NOT allowed to use APOC to load CSV or create nodes/relationships. You must use plain Cypher (LOAD CSV + MERGE/CREATE) because the Cypher code contributes marks. You can only use APOC in your two self-designed queries (Task 4)."

apoc.import.csv syntax

APOC import CSV — nodes and relationships together
CALL apoc.import.csv( [ // Node CSV files {filename: "file:/people.csv", labels: ["Person"]}, {filename: "file:/movies.csv", labels: ["Movie"]} ], [ // Relationship CSV files {filename: "file:/acted_in.csv", type: "ACTED_IN"} ], {} // config options )
✅ This replaces multiple individual LOAD CSV statements. People.csv → Person nodes. Movies.csv → Movie nodes. Acted_in.csv → ACTED_IN relationships. All in one CALL.

APOC also supports JSON import

APOC — load JSON data stream
CALL apoc.load.json("file:///data.json") YIELD value RETURN value

Dynamic Relationships with APOC

📖 The Problem — Multiple Relationship Types from One CSV

Imagine your CSV has people connected by different relationship types in the same file: some are FRIENDS, some are COLLEAGUES, some are SIBLINGS. Without APOC, you'd need a separate CSV for each type. With APOC, you can read the relationship type from a column value and create it dynamically.

Without APOC — the painful way

Without APOC — need separate CSV files for each relationship type
// Need: people.csv, knows.csv, friends_with.csv, enemies_of.csv, ... // One LOAD CSV query per file = lots of repeated code

With APOC — read relationship type from a column

apoc.create.relationship — create relationships dynamically from CSV
LOAD CSV WITH HEADERS from "file:///people.csv" AS row MERGE (p1:Person {name: row.node1}) MERGE (p2:Person {name: row.node2}) WITH p1, p2, row CALL apoc.create.relationship(p1, row.relationship, {}, p2) YIELD rel RETURN rel
✅ The CSV has columns: node1, node2, relationship. For each row:
1. Find or create Person p1 (from node1 column)
2. Find or create Person p2 (from node2 column)
3. Create a relationship of whatever type is in the "relationship" column (e.g. KNOWS, FRIENDS_WITH, ENEMIES_OF)
One CSV file, many relationship types — no repeated code needed!
node1 node2 relationship
Alice Bob KNOWS
Alice Carol FRIENDS_WITH
Bob Dave ENEMIES_OF

Example people.csv — APOC reads the "relationship" column and creates the correct relationship type for each row.

APOC Text Functions

📖 Why APOC Text Functions?

Regular Cypher has basic string functions like STARTS WITH, CONTAINS, toUpper(), toLower(). APOC adds many more text manipulation functions for cleaning and transforming string data during ETL.

APOC Function What it does Example Result
apoc.text.capitalize(s) Capitalise the first letter of a string apoc.text.capitalize("hello") "Hello"
apoc.text.capitalizeAll(s) Capitalise the first letter of every word apoc.text.capitalizeAll("hello world") "Hello World"
apoc.text.trim(s) Remove leading and trailing whitespace apoc.text.trim(" alice ") "alice"
apoc.text.decapitalize(s) Lowercase the first letter apoc.text.decapitalize("Alice") "alice"
apoc.text.replace(s, regex, to) Replace text using a regex pattern apoc.text.replace("hello world", "o", "0") "hell0 w0rld"
apoc.text.split(s, delim) Split a string into a list by delimiter apoc.text.split("a,b,c", ",") ["a","b","c"]
apoc.text.join(list, delim) Join a list of strings into one string apoc.text.join(["a","b"], ",") "a,b"

Using APOC text functions in a query

Clean up data — capitalise all person names
MATCH (p:Person) RETURN apoc.text.capitalizeAll(p.name) AS cleanName

Variable-Length Paths in Cypher

📖 The Problem

Writing a path of length 10 in Cypher looks terrible: (a)-->()-->()-->()-->()-->()-->()-->()-->()-->(b). Variable-length path syntax solves this by using the asterisk * to specify path length.

Variable-Length Path Syntax

(a)-[*5]->(b)
Exactly 5 hops from a to b
(a)-[*..5]->(b)
Any path from 0 to 5 hops (0 to 5 inclusive)
(a)-[*5..]->(b)
5 or more hops (no upper limit)
(a)-[*3..5]->(b)
Between 3 and 5 hops inclusive
(a)-[*]->(b)
Any number of hops (no limit — use with caution!)

Real example — find all people within 2 hops

Variable length path — find people 1 or 2 hops away
// Without variable length (2 hops, longwinded) MATCH (alice:Person {name:"Alice"})-->()-->(other) RETURN other // With variable length (1 to 2 hops, clean) MATCH (alice:Person {name:"Alice"})-[*1..2]->(other) RETURN other

Finding the shortest path

shortestPath — find the shortest connection between two nodes
MATCH (alice:Person {name:"Alice"}), (bob:Person {name:"Bob"}) MATCH path = shortestPath((alice)-[*]->(bob)) RETURN path, length(path)
✅ In plain English: "Find the shortest path from Alice to Bob following any relationship type."
Returns the path and how many hops it takes. Real-world use: find the shortest connection in a social network (like LinkedIn's "2nd degree connections").
⚠️ Warning — Unbounded Paths are Dangerous

Using [*] (no limit) on a large graph can take forever — it traverses the entire reachable graph. Always set an upper bound like [*..10] unless you're using shortestPath (which stops at the first match).

APOC Path Expander

📖 Why APOC Path Expander?

Cypher's variable-length paths [*] are simple but limited. APOC's apoc.path.expand gives much more control: you can filter by relationship type WITH direction, whitelist/blacklist specific node labels, and specify start and end conditions.

apoc.path.expand — the basic syntax

apoc.path.expand — syntax
MATCH (start:Person {name: "Praveena"}) CALL apoc.path.expand( start, // starting node "KNOWS", // relationship filter (type + direction) null, // label filter (null = no filter) 1, // minimum hops 2 // maximum hops ) YIELD path RETURN path, length(path) AS hops ORDER BY hops
✅ In plain English: "From Praveena, follow KNOWS relationships between 1 and 2 hops. Return each path and its length, sorted by length."

Label Filter Prefixes — controlling which nodes to include

Prefix Meaning Example
+ Whitelist — only traverse nodes WITH this label "+Engineering" — only follow paths through Engineering nodes
- Blacklist — skip nodes WITH this label "-Field" — don't traverse through Field nodes
/ End node — stop the path when a node with this label is reached "/Engineering" — terminate path at Engineering nodes
> End node (end here, include in result) ">Engineering" — paths that end AT Engineering nodes

Examples with label filters

Example 1 — whitelist: only Engineering nodes (+ prefix)
MATCH (p:Person {name: "Praveena"}) CALL apoc.path.expand(p, "KNOWS", "+Engineering", 1, 2) YIELD path RETURN path, length(path) AS hops ORDER BY hops // Returns ONLY paths through Engineering-labelled nodes
Example 2 — blacklist: exclude Field nodes (- prefix)
MATCH (p:Person {name: "Praveena"}) CALL apoc.path.expand(p, "KNOWS", "-Field", 1, 2) YIELD path RETURN path, length(path) AS hops ORDER BY hops // Returns paths EXCLUDING Field nodes
Example 3 — multiple relationship types with direction + end label
MATCH (p:Person {name: "Alicia"}) CALL apoc.path.expand(p, "FOLLOWS>|KNOWS", "/Engineering", 1, 3) YIELD path RETURN path, length(path) AS hops ORDER BY hops
✅ In plain English: "From Alicia, follow outgoing FOLLOWS or any KNOWS relationships, up to 3 hops, stopping the path as soon as an Engineering-labelled node is reached."
The > in FOLLOWS> means outgoing direction only. The | means OR. The /Engineering terminates paths at Engineering nodes.

Cypher path vs APOC path — when to use which

Cypher variable path [*]
  • Simple, built-in
  • Can filter by relationship type
  • All relationships treated equally (no per-type direction)
  • No label filtering during traversal
  • Use for: simple multi-hop queries
APOC apoc.path.expand
  • More control — direction per relationship type
  • Whitelist/blacklist node labels
  • Specify end nodes for traversal
  • More complex syntax (CALL + YIELD)
  • Use for: complex traversal with label/direction constraints

Virtual Nodes & Relationships

📖 What are Virtual Nodes and Relationships?

Virtual nodes and relationships are created in memory only — they are NOT stored in the Neo4j database. They are generated on-the-fly for a query result and exist only for that query's duration. Use them to represent computed or summarised connections for visualisation.

Why use virtual nodes and relationships?

Imagine you want to show "which students have the same understanding level." To check this, you'd have to traverse: Student → TestScore → Level ← TestScore ← Student — a long complex path. Instead, you can create a virtual SIMILAR_LEVEL relationship between the two students directly, which makes the visualisation much cleaner without permanently storing that relationship in the database.

apoc.create.vRelationship — virtual relationship
MATCH (s1:Student)-[:HAS]->(:TestScore)-[:ASSIGNED_TO]->(l:Level)<-[:ASSIGNED_TO]-(:TestScore)<-[:HAS]-(s2:Student) CALL apoc.create.vRelationship(s1, 'SIMILAR_LEVEL', {level: l.level}, s2) YIELD rel RETURN s1, rel, s2
✅ In plain English: "Find pairs of students who have the same test score level. Create a virtual SIMILAR_LEVEL relationship between them (with the level as a property). Show it in the visualisation — but don't store it in the database."
The SIMILAR_LEVEL relationship disappears after the query ends — it was never written to disk.

Virtual Nodes — apoc.create.vNode

apoc.create.vNode — virtual node
CALL apoc.create.vNode(['Person'], {name: 'Virtual Alice'}) YIELD node RETURN node // Creates a node that appears in the result but is NOT stored in the database

Lecture 10 — Key Takeaways

  • Cypher uses implicit grouping — no GROUP BY keyword. Non-aggregated expressions in RETURN are the grouping keys automatically.
  • COUNT(x) counts non-null values. COUNT(*) counts all rows including nulls. COUNT always returns 0 (never NULL) if no matches.

  • LIMIT restricts how many rows are returned. SKIP skips the first N rows. Together they do pagination.
  • UNWIND turns a list back into individual rows (opposite of COLLECT).
  • List comprehension: [x IN list WHERE condition | expression] — create a new list from an existing one with optional filtering.
  • Pattern comprehension: create a list from a graph pattern inline in the RETURN clause.
  • APOC = Awesome Procedures on Cypher. 450+ library functions. Exam: understand what it does, not the syntax.
  • APOC import CSV: apoc.import.csv() imports nodes and relationships together in one call.
  • APOC dynamic relationships: apoc.create.relationship() reads relationship type from a CSV column value.
  • Variable length paths: [*5]=exact, [*..5]=0-5, [*5..]=5+, [*3..5]=3-5.
  • APOC path expander: apoc.path.expand() with +label (whitelist), -label (blacklist), /label (end when reached), direction per relationship.
  • Virtual nodes/relationships exist only in memory — not stored in DB. Used for visualisation of computed connections.
