01

Date / Time Dimensions

📖 Why Use a Date Dimension?

Almost every data warehouse has a date dimension. Instead of storing a SQL timestamp, a date dimension table stores rich date properties that can't be computed from a timestamp (holiday/non-holiday, selling season, fiscal quarter, weekday/weekend).

Typical Grain: One Row = One Day

For finer time measurements (hours, minutes), use a separate time-of-day dimension rather than cramming it into the date dimension. Why? 20 years of daily data = ~7,300 rows. 20 years of minute-by-minute = over 10 million rows.

Why Not Just Use SQL Timestamps?

  • Timestamps don't tell you if a day is a holiday, weekend, or fiscal quarter
  • Date dimension avoids relying on special SQL date functions
  • Makes use of indices for faster filtering
  • Allows pre-computed flags for reporting (e.g., "Holiday" not just "1")

Use Meaningful Labels, Not 0/1

🎯 Exam Point — Always Use Meaningful Values

Use "Holiday" / "Non-Holiday" not 1 / 0. Use "Weekday" / "Weekend" not Y / N. Why? Because the data warehouse is for managers, not developers. When a value appears in a Power BI report, "Holiday" is instantly understandable; "1" is not.

Multiple Hierarchies from One Date Dimension

One date dimension table can support multiple concept hierarchies for different purposes:

  • Date → Month → Quarter → Year (calendar reporting)
  • Date → Week → Year (ISO week reporting)
  • Holiday / Non-Holiday (for promotion analysis)
  • Weekday / Weekend (for behaviour analysis)

Having multiple hierarchies from one table makes the DW more flexible and allows more types of roll-up/drill-down.

02

Surrogate Keys

📖 Surrogate Key vs. Natural Key

Natural key: A meaningful identifier from the real world (e.g., passport number, student ID, driver's licence). Surrogate key: A meaningless integer assigned by the data warehouse (e.g., 1, 2, 3, 4...).

Why Use Surrogate Keys in DW Dimension Tables?

  • Insulation from operational system changes: If a customer's email address (natural key) changes, historical fact table links are not broken because the surrogate key stays the same.
  • Easy multi-source integration: Different systems may use different IDs for the same entity. Surrogate keys provide a unified identifier.
  • Smaller storage: A 4-byte integer surrogate key is much smaller than an 8-character passport number. Fact tables referencing dimension keys become thinner → better performance.
  • Handle exceptional cases: What if someone doesn't have a driver's licence? You can't use a NULL natural key. A surrogate key allows you to assign a special value like "Unknown" (surrogate key = 0).
🎯 Exam Trap — Natural Keys Are Risky

If you use email address as the PK and a customer changes their email, you lose the link to all their historical sales records. Surrogate keys prevent this — they never change, even if the person's natural ID changes.

Degenerate Dimension

📖 Definition

A degenerate dimension is a dimension key that exists in the fact table but has NO corresponding dimension table. Examples: Transaction ID, Invoice Number. These are identifiers with no descriptive attributes worth storing in a separate table.

Two options for degenerate dimensions: (1) Discard it if not needed for analysis. (2) Keep it directly in the fact table as a foreign key without creating a dimension table. Used for grouping line items (e.g., "all products in the same shopping cart share the same Transaction ID").

03

How Many Dimensions? (Performance)

The number of dimension tables and how you group attributes affects storage size and query performance significantly. Three design options were compared:

OptionDesignStoragePerformanceModern Use
AOne "Everything" dimension — all 100 attributes in 1 table~200 GB (huge)Fastest (no joins)Popular for cloud, ML, real-time analytics
BEach attribute gets its own dimension table (100 tables)~21.2 GBSlow (100 joins)Not recommended
C5 logical dimensions grouping correlated attributes~2.8 GB (smallest)Good (few joins)Traditional best practice
🎯 Exam Point — Why Option C Was Traditionally Best

Correlated attributes (e.g., product name and brand) should be in the same dimension because they have the same number of distinct combinations as products. Uncorrelated attributes (e.g., product and date) produce explosive combinations if combined, making Option A's table massive. Option C groups correlated attributes = compact dimensions.

💡 Modern Shift: Option A is Now Common

With cloud storage (cheap) and columnar databases (only scan needed columns), Option A ("One Big Table") is now popular for: real-time analytics, machine learning (one row = one feature vector), BI tools (Power BI/Tableau prefer flat single tables). The shift = from storage-optimised (Option C) to compute-optimised (Option A).

04

Slowly Changing Dimensions (SCD)

📖 Definition

A Slowly Changing Dimension (SCD) is a dimension where attribute values change occasionally over time (not frequently). Examples: customer address, store location, employee department, product category.

The question is: when an attribute value changes, do we overwrite the old value (losing history) or preserve both old and new (keeping history)?

Type 1 — Overwrite

What it does: Simply update the existing row with the new value.

  • ✅ Simple to implement
  • ✅ No extra rows
  • ❌ Loses all historical data
  • ❌ Historical reports change retroactively

Use when: History doesn't matter (e.g., phone number).

Type 2 — New Row

What it does: Add a NEW row for the new value. Old row is flagged as expired; new row is flagged as current.

  • ✅ Full history preserved
  • ✅ Accurate historical reporting
  • ❌ Table grows over time
  • ❌ Requires surrogate keys

Use when: History matters (e.g., customer address, district boundaries).

Type 3 — New Column

What it does: Add a new column to store the old value (e.g., "Previous District" + "Current District").

  • ✅ No new rows — single row per entity
  • ✅ Can report both old and new
  • ❌ No timestamps — can't tell when change happened
  • ❌ Schema gets complicated with many changes

Use when: Only 1–2 previous values need tracking.

🎯 Exam Comparison — Which Type to Choose?

Ask yourself: "Does the historical value matter for reporting?"
— Phone number changed? Type 1 (history unimportant, analysts don't care which number was active).
— Customer moved from Crawley to Claremont? Type 2 (historical sales must be attributed to correct location).
— District boundaries reorganised? Type 2 or Type 3 (need to compare old vs. new).
Type 2 requires surrogate keys — because the natural key may be the same for old and new rows.

05

Dimension Roles

📖 Definition

A dimension role occurs when a single dimension table is used in multiple ways within the same fact table. The same physical table plays different "roles" using separate foreign keys.

Airline Example

A flight has: Departure City, Arrival City, Departure Time, Arrival Time. Instead of creating separate "Departure Location" and "Arrival Location" dimension tables (which would be identical), you create ONE location dimension and reference it TWICE in the fact table — once as departure_city_key and once as arrival_city_key.

  • ONE Date dimension table, but TWO foreign keys in the fact table: order_date_key, ship_date_key
  • ONE Location dimension table, but TWO foreign keys: origin_location_key, destination_location_key
💡 Why This Is Better

Creating two separate "Departure Time" and "Arrival Time" dimension tables would be redundant — they'd have identical structure and data. One table with two references is cleaner, smaller, and consistent.

06

Junk Dimension

📖 Definition

A junk dimension is a single dimension table that groups together several low-cardinality, miscellaneous flags or indicators that don't fit into any existing dimension. This avoids cluttering the fact table with many small single-column dimensions.

Examples of attributes that become junk dimensions: payment method (Cash / Credit Card / Check), bag type (Paper / Plastic / None), order type (Standard / Express / Next-Day).

When to Use a Junk Dimension

  • Attributes are low in cardinality (few possible values)
  • Attributes are few in number (not too many different attributes)
  • Attributes are somewhat related in usage
  • They don't logically belong to any existing dimension

Transactional vs. Snapshot Fact Tables

FeatureTransactionalSnapshot
What one row representsA single discrete event (e.g., one deposit, one withdrawal)The state at a specific point in time (e.g., end-of-day balance)
Detail levelMaximum — every individual event recordedSummarised — state at predefined intervals (daily, weekly, hourly)
Update frequencyEvery time an event occursAt predefined intervals (daily snapshot, weekly snapshot)
Use caseAnalysing individual events, auditing, detailed reportingTracking trends over time, inventory levels, account balances
ExamplesEvery debit/credit transaction to a bank accountEnd-of-day account balance for each day
🎯 Exam Practice — Converting Transactional to Snapshot

If given a transactional table (all individual debits/credits), you must be able to construct the corresponding daily snapshot table. Key rule: if no transactions occur on a day, the balance STAYS THE SAME as the previous day. The snapshot carries forward the last known value.

💡 Snapshot Example

Jim's transactions: Mar 20: +20, −5 = balance +15. Mar 21: no transactions = balance stays 15. Mar 22: −10 = balance 5.
Snapshot table rows: (Jim, Mar 20, 15), (Jim, Mar 21, 15), (Jim, Mar 22, 5).

08

International Issues

Multiple Currencies

When an organisation operates across countries, each transaction may be in a different currency. Solution: store each currency-denominated fact in TWO columns — one in local currency, one in a standard currency (e.g., USD). Add a Currency dimension to indicate which currency the local column uses. Use the historical exchange rate at the time of the transaction.

Time Zones

Decide: local time or standardised time (GMT/UTC)?

  • Local time: Better when local business patterns matter (e.g., a café's busiest hours are 8am–10am locally, not midnight UTC)
  • Standardised time (GMT): Better when relative ordering of global events matters (e.g., stock trades must be in sequence globally)
★

Summary

Lecture 04 — Key Takeaways

  • Date dimension: one row per day, rich attributes (holiday, weekday, fiscal year), use meaningful labels not 0/1.
  • Separate date and time dimensions to avoid row count explosion.
  • Surrogate key = meaningless integer assigned by DW. Better than natural keys for: insulation from change, multi-system integration, smaller storage, exceptional cases.
  • Degenerate dimension = dimension key in fact table with no dimension table (e.g., transaction ID).
  • SCD Type 1 = overwrite (simple, loses history). Type 2 = new row (preserves history, needs surrogate key). Type 3 = new column (limited history, no timestamp).
  • Type 2 requires surrogate keys to track different versions of the same entity.
  • Dimension roles: one dimension table referenced multiple times via different FKs (e.g., date dimension used as order_date and ship_date).
  • Junk dimension: groups miscellaneous low-cardinality flags into one table.
  • Transactional: every event. Snapshot: state at predefined intervals (carries forward if no event).

Exam-Style Practice Questions

A customer named Alice moves from Crawley to Claremont. Explain how SCD Type 1 and Type 2 would each handle this change. Which should you choose and why?

Hint: Type 1 = overwrite Crawley with Claremont (old sales now attributed to Claremont — incorrect!). Type 2 = add new row for Claremont, keep Crawley row as expired. Type 2 is correct — location matters for historical sales attribution.

A WA airline fact table needs to record both departure and arrival airports for each flight. Should you create two separate airport dimension tables or one? Explain.

Hint: One airport dimension table. Two foreign keys in the fact table: departure_airport_key and arrival_airport_key. This is a "dimension role" — same table, different roles. Creating two identical tables is redundant.

Explain why a bank account balance is a semi-additive fact. Give an example of a calculation that IS valid and one that is NOT.

Hint: Valid: sum balances across accounts (total money in the bank). Invalid: sum balance on Monday + Tuesday + Wednesday (double-counting — the same money counted multiple times).

Given the transactional table below, construct the daily snapshot for Jim and Lucy assuming starting balance = 0. Transactions: Jim Mar 20 +20, Jim Mar 20 -5, Lucy Mar 20 +40, Jim Mar 22 -10, Lucy Mar 22 -5, Lucy Mar 22 +10.

Hint: Jim Mar 20=15, Mar 21=15 (no transactions), Mar 22=5. Lucy Mar 20=40, Mar 21=40 (no transactions), Mar 22=45.

What is a surrogate key? Give three reasons why it is preferred over a natural key (like a passport number) as the primary key in a dimension table.

Hint: Surrogate = meaningless integer. Reasons: (1) insulation — natural key can change; (2) integration — different source systems use different IDs; (3) storage — integer is smaller than full passport number; (4) exceptional cases — what if someone has no passport?

What is a "junk dimension"? Give an example of when you would create one.

Hint: Groups miscellaneous low-cardinality flags (payment method = Cash/Card/Check; bag type = Paper/Plastic/None). Use when these attributes don't logically belong to any existing dimension and would otherwise clutter the fact table.
